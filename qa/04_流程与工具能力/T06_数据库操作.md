## 1. 概述

数据库操作是游戏QA进行数据校验、问题定位、数据修复的重要技能。游戏项目通常涉及MySQL/PostgreSQL等关系型数据库、Redis缓存、MongoDB文档数据库等多种存储方案。QA需要掌握SQL查询与优化、数据一致性校验、安全操作规范等能力，在不影响线上数据安全的前提下高效完成数据验证与问题排查。

## 2. 知识体系

### 2.1 SQL优化(执行计划与索引)

**EXPLAIN执行计划关键字段：**

| 字段 | 含义 | 关注点 |
|------|------|--------|
| type | 访问类型 | ALL(全表扫描)→index→range→ref→const，越靠右越好 |
| key | 实际使用的索引 | NULL表示未使用索引 |
| rows | 预估扫描行数 | 越小越好 |
| Extra | 额外信息 | Using filesort/Using temporary需优化 |

**索引优化原则：**
- 最左前缀原则：联合索引(a,b,c)，查询条件需从a开始
- 覆盖索引：查询字段全在索引中，避免回表
- 避免索引失效：不在索引列上做函数运算、隐式类型转换、LIKE以%开头
- 选择性原则：高选择性列(如user_id)适合建索引，低选择性列(如gender)不适合

**常用优化技巧：**
```sql
-- 避免SELECT *
SELECT id, name, level FROM player WHERE server_id = 1;

-- 分页优化：避免大偏移量
-- 慢：SELECT * FROM log LIMIT 1000000, 20;
-- 快：SELECT * FROM log WHERE id > 1000000 LIMIT 20;

-- 批量查询替代循环单条查询
SELECT * FROM item WHERE player_id IN (1001, 1002, 1003);

-- 避免在WHERE中使用OR，改用UNION ALL
SELECT * FROM player WHERE level = 60
UNION ALL
SELECT * FROM player WHERE vip_level = 10;
```

### 2.2 事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 | 游戏场景 |
|---------|------|-----------|------|------|---------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 最高 | 几乎不用 |
| READ COMMITTED | 不会 | 可能 | 可能 | 高 | 日志查询 |
| REPEATABLE READ | 不会 | 不会 | 可能(InnoDB通过MVCC避免) | 中 | MySQL默认，大多数场景 |
| SERIALIZABLE | 不会 | 不会 | 不会 | 最低 | 交易/充值等强一致场景 |

**游戏中常见事务问题：**
- 背包操作并发：两个请求同时扣减同一物品 → 需要行锁/乐观锁
- 交易系统：A给B转金币，需保证原子性 → 事务+行锁
- 排行榜更新：高并发写入 → 考虑异步批量更新

### 2.3 Redis数据结构与应用场景

| 数据结构 | 命令示例 | 游戏应用场景 |
|---------|---------|-------------|
| String | `GET/SET/INCR` | 玩家Session、配置缓存、计数器 |
| Hash | `HGET/HSET/HGETALL` | 玩家属性、装备信息 |
| List | `LPUSH/RPOP/LRANGE` | 消息队列、聊天记录 |
| Set | `SADD/SMEMBERS/SINTER` | 好友列表、在线玩家集合 |
| Sorted Set | `ZADD/ZRANGE/ZRANK` | 排行榜、延迟任务队列 |
| Bitmap | `SETBIT/GETBIT/BITCOUNT` | 签到记录、活跃统计 |

**Redis常用调试命令：**
```bash
# 连接
redis-cli -h host -p 6379 -a password

# 查看key信息
TYPE key_name                    # 查看类型
TTL key_name                     # 查看过期时间
OBJECT ENCODING key_name         # 查看编码方式

# 模糊查找(生产环境禁用KEYS，用SCAN)
SCAN 0 MATCH "player:*" COUNT 100

# 内存分析
MEMORY USAGE key_name            # 单个key内存占用
INFO memory                      # 整体内存信息

# 慢查询
SLOWLOG GET 10                   # 查看最近10条慢查询
```

### 2.4 MongoDB聚合管道

**常用聚合阶段：**

| 阶段 | 作用 | 示例 |
|------|------|------|
| $match | 过滤文档 | `{$match: {server_id: 1}}` |
| $group | 分组聚合 | `{$group: {_id: "$level", count: {$sum: 1}}}` |
| $sort | 排序 | `{$sort: {count: -1}}` |
| $project | 字段投影 | `{$project: {name: 1, level: 1}}` |
| $lookup | 关联查询 | 类似SQL的JOIN |
| $unwind | 展开数组 | 将数组字段拆分为多条文档 |
| $limit/$skip | 分页 | 限制/跳过文档数 |

**游戏数据分析示例：**
```javascript
// 统计各等级玩家数量分布
db.players.aggregate([
  { $match: { server_id: 1, status: "active" } },
  { $group: { _id: "$level", count: { $sum: 1 } } },
  { $sort: { _id: 1 } }
]);

// 查询玩家最近7天充值总额
db.orders.aggregate([
  { $match: {
    create_time: { $gte: ISODate("2026-03-22") },
    status: "success"
  }},
  { $group: {
    _id: "$player_id",
    total: { $sum: "$amount" },
    count: { $sum: 1 }
  }},
  { $sort: { total: -1 } },
  { $limit: 100 }
]);
```

### 2.5 数据安全操作规范

**操作分级：**

| 级别 | 操作类型 | 审批要求 | 备份要求 | 示例 |
|------|---------|---------|---------|------|
| L1-只读 | SELECT查询 | 无需审批 | 无需备份 | 数据校验、问题排查 |
| L2-低风险写 | INSERT(测试数据) | 组长审批 | 建议备份 | 创建测试账号 |
| L3-中风险写 | UPDATE(少量数据) | 主管审批 | 必须备份 | 修复玩家数据 |
| L4-高风险写 | DELETE/批量UPDATE | 总监审批+DBA执行 | 必须备份+验证 | 数据清理、批量修复 |

**安全操作铁律：**
1. 生产环境操作前必须先在测试环境验证SQL
2. UPDATE/DELETE必须带WHERE条件，先SELECT确认影响范围
3. 批量操作必须限制条数(LIMIT)，分批执行
4. 操作前备份相关数据，操作后验证结果
5. 所有生产操作记录操作日志(谁/何时/做了什么/影响范围)

```sql
-- 安全操作模板
-- Step 1: 备份
CREATE TABLE player_backup_20260329 AS
SELECT * FROM player WHERE id IN (1001, 1002);

-- Step 2: 确认影响范围
SELECT id, gold FROM player WHERE id IN (1001, 1002);

-- Step 3: 执行修改
UPDATE player SET gold = gold + 1000
WHERE id IN (1001, 1002)
LIMIT 2;

-- Step 4: 验证结果
SELECT id, gold FROM player WHERE id IN (1001, 1002);
```

## 5. 执行流程

### 5.1 准备阶段
- 确认数据库连接信息与权限(只读/读写)
- 了解数据库表结构与数据模型
- 准备查询SQL并在测试环境验证
- 确认操作审批流程(写操作需审批)
- 备份相关数据(写操作前必须)

### 5.2 执行阶段
- 按照安全操作规范执行SQL
- 写操作分批执行，每批验证结果
- 记录操作日志(SQL、执行时间、影响行数)
- 异常情况立即停止并评估影响
- 操作完成后验证数据正确性

### 5.3 分析阶段
- 对比操作前后数据变化
- 验证关联数据的一致性
- 检查缓存是否需要同步更新
- 分析慢查询与性能瓶颈
- 评估数据修复的完整性

### 5.4 报告阶段
- 输出数据校验报告(一致性/完整性)
- 记录数据修复操作日志
- 汇总数据库性能分析结果
- 提出表结构/索引优化建议
- 更新数据操作知识库

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：UPDATE忘记加WHERE条件导致全表更新**
- 现象：执行UPDATE后发现所有玩家数据被修改
- 原因：SQL语句遗漏WHERE条件，或WHERE条件写错
- 解决：强制要求先SELECT确认范围，开启sql_safe_updates模式，写操作必须Code Review

**坑2：Redis缓存与DB数据不一致**
- 现象：客户端显示的数据与DB查询结果不同
- 原因：修改DB后未同步更新/清除Redis缓存
- 解决：数据修复后同步清除相关Redis缓存，了解缓存更新机制(Cache Aside/Write Through)

**坑3：生产环境执行KEYS命令导致Redis阻塞**
- 现象：执行`KEYS *`后Redis响应变慢，影响线上服务
- 原因：KEYS命令会遍历所有key，O(N)复杂度，阻塞主线程
- 解决：生产环境禁用KEYS命令，使用SCAN命令替代，设置rename-command

**坑4：大表查询未加LIMIT导致内存溢出**
- 现象：查询日志表时客户端卡死或OOM
- 原因：日志表数据量巨大(千万级)，SELECT无LIMIT返回全部数据
- 解决：所有查询必须加LIMIT，大数据量使用分页查询，导出用mysqldump

**坑5：时区问题导致数据查询结果错误**
- 现象：按时间范围查询的数据与预期不符
- 原因：数据库时区、应用时区、查询工具时区不一致
- 解决：统一使用UTC存储，查询时明确时区转换，`SET time_zone = '+8:00'`
