# AI辅助测试

> 所属板块: 流程与工具能力 | 编号: T07 | 最后更新: 2026-03-30

---

## 1. 概述

AI辅助测试是利用大语言模型（LLM）能力赋能测试全流程的新兴方向，核心价值在于降低自动化门槛、提升用例生成效率、让非技术背景人员参与自动化建设。关键设计原则是**"AI只生成不执行"**——让AI做它最擅长的事（理解意图、生成代码），让确定性程序做它最擅长的事（稳定执行）。本Skill以Microsoft Edge QA团队开源的AutoGenesis项目为核心参考案例，覆盖AI+MCP架构设计、代码生成机制、跨平台落地实践。

## 2. 知识体系

### 2.1 AI在测试中的三种应用路径

| 路径 | 方式 | 优点 | 缺点 | 适用阶段 |
|------|------|------|------|---------|
| AI直接执行测试 | AI读取截图/DOM，实时决策操作 | 零代码 | 不确定性高、结果不可靠、不可回溯 | Demo/探索 |
| AI生成+人工执行 | AI生成用例文本，人工转化为脚本 | 降低设计成本 | 仍需编码能力 | 过渡期 |
| **AI生成+程序执行** | AI生成可执行代码，确定性框架执行 | 门槛低、稳定、可审查 | 需架构设计 | **生产推荐** |

### 2.2 为什么不让AI直接执行测试

**执行过程不可控：**
- 长任务中AI容易"走神"，忘记上下文、重复步骤
- 遇到意外弹窗/页面变化可能陷入无限循环
- 无法准确追踪"当前执行到哪一步"，出错后难以定位

**结果判断不可靠：**
- AI基于截图判断"测试是否通过"存在模糊性
- 传统断言可精确验证文本/属性/数值，AI只能给出主观判断
- 判断过程是黑盒，失败时无法追溯判断逻辑

### 2.3 AI+MCP四层架构（AutoGenesis方案）

```
┌─────────────────────────────────────────────┐
│  用户层 — Gherkin BDD 自然语言描述测试场景     │  不依赖AI
├─────────────────────────────────────────────┤
│  LLM层 — 将自然语言翻译为MCP工具调用序列       │  ★ AI仅在此层工作
│         生成Python步骤定义代码                 │
├─────────────────────────────────────────────┤
│  MCP Server层 — 跨平台能力统一封装             │  不依赖AI
│  PyWinauto(Windows) / Appium(macOS/iOS/Android)│
├─────────────────────────────────────────────┤
│  执行层 — Behave BDD框架确定性执行             │  不依赖AI
└─────────────────────────────────────────────┘
```

关键原则：**AI的工作边界严格限定在第二层（LLM层），执行层完全不依赖AI。**

### 2.4 MCP协议在测试中的作用

MCP（Model Context Protocol）是AI与外部工具之间的标准化通信协议：
- 将平台自动化能力封装为标准化工具集（点击、输入、断言、截图等）
- AI通过MCP协议调用工具，而非直接操作设备
- 不同平台实现不同的MCP Server，对AI层暴露统一接口
- 约束AI行为边界，防止AI执行未授权操作

### 2.5 BDD + AI 代码生成工作流

**Step 1：用Gherkin写测试场景（接近自然语言）**
```gherkin
Feature: 游戏商城购买流程
  Scenario: 购买钻石礼包
    Given 用户已登录且在主界面
    When 用户打开商城页面
    And 用户选择"钻石礼包"商品
    And 用户点击购买并确认支付
    Then 钻石数量增加对应数额
    And 背包中出现礼包道具
```

**Step 2：AI生成Python步骤定义代码**
- AI将每个Given/When/Then步骤翻译为MCP工具调用序列
- 生成标准的Behave步骤定义文件（steps/*.py）

**Step 3：确定性执行**
- 运行 `behave features/` 执行测试
- 执行过程零AI调用，完全确定性

### 2.6 代码生成安全机制：Preview-Confirm

```
Step 1 — before_gen_code(feature_file, step_file)  → 预览将要修改的文件
Step 2 — gen_code(code_content)                     → AI生成代码，人工审查
Step 3 — confirm_gen_code()                         → 确认后写入文件
```

三阶段机制确保：
- 人类始终拥有代码写入的最终控制权
- AI生成的代码必须经过审查才能落地
- 防止AI"乱写"代码导致测试逻辑错误

### 2.7 跨平台MCP Server能力

**PyWinauto MCP Server（Windows桌面）：**
- 通过Windows UI Automation API和Win32 API访问控件结构
- 支持现代应用（UWP/WPF/WinUI）和传统应用（Win32/MFC）
- 工具集：应用管理、交互操作、鼠标高级操作、验证断言、代码生成

**Appium MCP Server（macOS/iOS/Android）：**
- Android通过UIAutomator2、iOS通过XCUITest访问控件树
- 通过FastMCP框架按平台注册工具，避免无关工具干扰AI选择
- 工具集：通用操作、验证断言、代码生成 + 各平台专属工具

### 2.8 团队分工模型

| 角色 | 职责 | 技能要求 |
|------|------|---------|
| FTE工程师 | 构建工具链、制定标准、培训团队、维护MCP Server | 编程能力、架构设计 |
| QA人员（含非技术背景） | 用自然语言写测试场景、审查AI生成代码、执行测试 | 业务理解、用例设计 |
| AI（LLM） | 将Gherkin翻译为MCP工具调用、生成步骤定义代码 | — |

核心价值：让非技术人员用自己擅长的方式（自然语言描述测试场景）直接产出专业级自动化代码。

## 5. 执行流程

### 5.1 准备阶段
- **参考项目特性档案**（[G00_项目特性](../00_项目特性/G00_项目特性.md)），确认GM指令集合（用于BDD Given步骤）
- 搭建MCP Server环境（PyWinauto/Appium按平台选择）
- 配置LLM接入（GitHub Copilot或其他LLM服务）
- 安装Behave BDD框架及依赖
- 培训团队Gherkin语法（Given/When/Then）
- 建立代码审查规范（Preview-Confirm流程）

### 5.2 执行阶段
- 用Gherkin编写测试场景（.feature文件）
- 触发AI生成步骤定义代码（Send to Copilot / autoGenesis-run skill）
- 人工审查生成代码，确认写入
- 运行 `behave features/` 确定性执行（零AI调用）
- 收集执行日志、截图、断言结果

### 5.3 分析阶段
- 区分真实Bug与生成代码问题
- 生成代码有误 → 重新触发AI生成或手动修正
- 真实Bug → 提交缺陷管理系统
- 统计AI生成代码的一次通过率，持续优化Prompt

### 5.4 报告阶段
- 生成Behave/Allure测试报告
- 统计AI辅助效率指标（编写时间、通过率、人工修正率）
- 输出团队能力提升数据（非技术人员贡献占比）
- 定期Review AI生成代码质量趋势

## 6. 数据指标

| 指标 | 定义 | 基线值 | 告警阈值 |
|------|------|--------|---------|
| AI生成代码一次通过率 | AI生成代码无需人工修改即可执行通过的比例 | ≥80% | <60% |
| 确定性执行通过率 | 生成代码在执行层的通过率 | ≥99% | <95% |
| 场景编写效率 | 单个测试场景从编写到可执行的时间 | ≤15min | >30min |
| 非技术人员贡献率 | 非技术背景人员提交的用例占总用例比例 | ≥30% | <10% |
| 人工修正率 | AI生成代码需要人工修改的比例 | ≤20% | >40% |
| 月执行步骤量 | 每月自动化执行的总步骤数 | 持续增长 | 连续下降 |
| 用例规模 | 可执行的自动化用例总数 | 持续增长 | 停滞 |
| AI调用成本 | 仅生成阶段产生，执行阶段为零 | 可控 | 超预算 |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：AI生成代码的元素定位策略不稳定**
- 现象：AI每次生成的定位方式不一致（有时用name，有时用xpath）
- 原因：LLM的不确定性导致生成策略漂移
- 解决：在MCP Server层制定定位优先级规则（AutomationId > Name > XPath），通过Prompt约束AI优先使用稳定定位器

**坑2：Gherkin描述过于模糊导致生成代码偏差**
- 现象：写"用户完成购买"，AI生成的步骤缺少关键验证
- 原因：自然语言描述缺乏足够细节
- 解决：制定Gherkin编写规范，要求每个步骤包含明确的操作对象和预期状态

**坑3：Behave同步框架与MCP异步实现的桥接问题**
- 现象：步骤执行超时或阻塞
- 原因：Behave是同步框架，MCP的Python实现是异步的
- 解决：使用janus.Queue实现线程桥接，call_tool_sync封装异步调用（超时400秒）

**坑4：非技术人员对AI生成代码缺乏审查能力**
- 现象：非技术人员直接Confirm所有生成代码，未发现逻辑错误
- 原因：代码审查需要一定技术背景
- 解决：建立分级审查机制——简单场景自审，复杂场景工程师复审

**坑5：MCP Server工具集过大导致AI选择错误工具**
- 现象：AI调用了不适合当前平台的工具
- 原因：所有平台工具混在一起，AI难以区分
- 解决：通过FastMCP按平台注册工具，只暴露当前平台相关的工具集
