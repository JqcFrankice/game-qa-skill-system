# 测试用例生成规范

> 所属板块: 流程与工具能力 | 编号: T08 | 最后更新: 2026-03-30
> 本Skill是所有AI测试用例生成的统一规范源，必须在生成任何用例前加载。

---

## 1. 概述

### 1.1 Skill定位

测试用例生成规范定义了AI生成游戏测试用例时的统一优先级(P0-P2)、编写规范、输出格式、回归分层和质量要求。本Skill是T04（测试设计）和T07（AI辅助测试）的桥梁，是所有testcase Skill的基础约束。

核心原则：**AI生成 + 人工校验 + 规则兜底**。

### 1.2 适用场景

- 所有AI测试用例生成任务
- 基于spec.md、design.md、proposal.md、tasks.md、缺陷分析报告生成测试点
- 系统类、玩法类、专项领域类测试用例输出

### 1.3 加载顺序

1. 先加载testcase-standards（本Skill）
2. 再按任务类型补充专项Skill
3. 若命中多个专项领域，可组合加载多个子Skill

### 1.4 输入资料优先级

若同时存在多类输入，按以下顺序理解需求：

1. spec.md
2. design.md
3. proposal.md
4. tasks.md
5. AI分析报告/缺陷上下文
6. 配置表/Excel/历史踩坑/UI稿/交互稿

### 1.5 目标

生成的测试用例必须满足：

- 覆盖核心功能路径与关键风险点
- 步骤清晰、可执行、可复现
- 预期结果具体、客观、可验证
- 输出结构统一，便于接口消费和XMind导入
- 产物可用于冒烟、功能回归和版本回归分层执行

## 2. 知识体系

### 2.1 优先级定义

#### 2.1.1 划分维度

用例优先级基于以下维度综合判定：

| 维度 | 说明 | 优先级方向 |
|------|------|------------|
| 功能路径重要度 | 距离用户完成核心目标越近，优先级越高 | 主干 > 分支 > 边缘 |
| 数据与资产风险 | 涉及奖励、扣费、排行、存档、状态同步时优先级更高 | 资产损失 > 数据异常 > 体验问题 |
| 影响用户范围 | 覆盖用户越多，优先级越高 | 全量 > 群组 > 个例 |
| 用户操作频率 | 操作频率越高，优先级越高 | 高频 > 中频 > 低频 |
| 恢复与追查成本 | 问题越难恢复、越难追查，优先级越高 | 不可恢复 > 可补偿 > 易恢复 |

**功能路径定义：**

- **主干功能路径**：用户完成产品核心目标必须经过的连续操作路径
- **分支功能路径**：从主干路径分岔出的可选操作，不影响主流程继续推进
- **边缘功能路径**：远离主流程、需主动深度操作才会进入的场景

**数据与资产风险定义：**

- **资产风险**：奖励重复发放、扣费异常、货币丢失、背包丢物、排行结算错误
- **状态风险**：状态错乱、重登回档、跨天异常、断线重连状态不一致
- **体验风险**：展示错误、音效问题、低频异常操作体验不佳

#### 2.1.2 优先级等级

**P0 - 核心可用性 / 资产安全**

| 项目 | 内容 |
|------|------|
| 测试目标 | 验证主流程可用、核心逻辑正确、资产与数据不出事故 |
| 失败影响 | 直接影响系统可用性、奖励发放、扣费正确性、数据完整性 |
| 典型场景 | 功能入口、开启条件、主流程、结算、奖励发放、消耗扣除、重登一致性 |
| 判定条件 | 命中以下任一项通常判为P0：主干路径、全量用户、资产风险高、不可恢复或高投诉风险 |

**P1 - 关键分支 / 状态正确性**

| 项目 | 内容 |
|------|------|
| 测试目标 | 验证主要分支、状态切换、界面展示、关键交互和联动正确性 |
| 失败影响 | 导致功能不完整、状态异常、明显体验问题或群体性投诉 |
| 典型场景 | 分支流程、边界条件、状态切换、列表展示、红点同步、邮件补发、联动系统验证 |
| 判定条件 | 分支路径、群组用户、中频操作、状态风险中高、回归影响较广 |

**P2 - 稳定性 / 边界 / 体验细节**

| 项目 | 内容 |
|------|------|
| 测试目标 | 验证弱网、中断、断线重连、极端边界、低频异常与体验细节 |
| 失败影响 | 触发概率相对较低，或主要影响稳定性、兼容性和边缘体验 |
| 典型场景 | 弱网、切后台、异常操作、极端输入、分辨率适配、音效、低频组合场景 |
| 判定条件 | 边缘路径、个例用户、低中频操作、可恢复问题或体验问题 |

#### 2.1.3 优先级使用规则

- 仅允许使用P0、P1、P2
- 若一条用例同时覆盖多个预期，取其中最高优先级
- 奖励发放、消耗扣除、排行结算、存档回档、重复发奖、扣费异常默认不低于P0
- 状态切换、关键分支、主要展示异常默认不低于P1
- 视觉细节、音效、低频极端体验问题通常归为P2

### 2.2 回归分层

#### 2.2.1 分层原则

- **冒烟回归**：只覆盖P0主流程和核心可用性
- **功能回归**：覆盖P0+P1，验证主要流程、关键分支、状态与联动
- **版本回归**：覆盖高风险P2稳定性、兼容性、异常与专项场景

#### 2.2.2 通用专项分层

若专项Skill未单独定义，默认按以下层级：

- **版本级回归一次**：分辨率、UI声音、返回键、通用弱网、中断、断线重连
- **系统首次接入必测**：红点联动、邮件补发、背包满转邮件、重登持久化、日志/错误码校验
- **高风险玩法每次回归**：奖励发放、消耗扣除、跨天重置、活动结束瞬间、排行结算、多端并发

### 2.3 编写规范

#### 2.3.1 五项基本要求

| 要求 | 定义 | 常见违规 |
|------|------|----------|
| 可读性 | 语言简洁明确，无歧义 | 读者需要猜测操作含义 |
| 可执行性 | 任何人按步骤执行都能得到一致结果 | 步骤模糊，执行结果不一致 |
| 独立性 | 每条用例尽量独立执行 | 依赖前一条用例结果才能执行 |
| 原子性 | 一条用例只验证一个核心点 | 一条用例验证多个无关点 |
| 可验证性 | 预期结果必须具体可见 | 只写"正常""成功""没问题" |

#### 2.3.2 前置条件规范

- 没有前置条件时填空字符串，不写"无"
- 必须写成可构造、可达到的明确状态
- 若需要特定账号、时间、配置、网络状态，必须写清楚
- 若依赖活动开关、灰度、GM配置、背包容量等，也应明确写出

正例：`账号完成新手引导且等级为10级，背包内已有道具"挑战券"x1，活动开关开启`
反例：`账号等级较高`

#### 2.3.3 测试步骤规范

- 每一步都必须是明确操作，不写抽象命令
- 涉及输入数据时，必须给出具体值
- 多步场景要体现真实执行顺序
- 若存在跨页面、重登、切后台、断网、时间调整等动作，必须明确发生时机

正例：
```
1. 登录账号A，进入活动主界面
2. 点击"领取奖励"按钮
3. 返回背包界面，查看奖励道具数量
```

反例：`测试领奖功能`

#### 2.3.4 预期结果规范

- 与步骤一一对应
- 优先写具体数值、提示文案、状态变化、界面元素变化
- 避免主观描述和泛化表达
- 信息不足时允许标注（待确认），并在附录汇总

正例：
```
1. 活动主界面展示奖励状态为"可领取"
2. 点击后弹出提示"领取成功"
3. 背包内"挑战券"数量从1增加到2
```

反例：`领取成功，功能正常`

#### 2.3.5 数值与状态要求

- 奖励、货币、次数、积分、经验等尽量写精确数值
- 边界值必须覆盖阈值前后
- 状态切换需体现"从哪里来，切到哪里去"
- 奖励与消耗需体现"是否展示、是否入账/扣除、失败是否回滚、重登后是否一致"

### 2.4 生成思路

在生成用例时，优先按以下顺序思考：

1. 先覆盖功能入口开关条件（配置表控制的开启/关闭）和主流程核心状态切换
2. 再覆盖关键分支、边界条件、奖励与消耗正确性；条件类功能需从配置表穷举所有条件类型
3. 再覆盖功能产出物的多场景显示一致性（装扮/头像/称号等外观类需逐场景验证展示入口）
   - 对"入口型功能"，不能只写"点入口进入"，必须同时覆盖：
     - 满足开启条件 → 入口显示/可点击
     - 不满足开启条件 → 入口隐藏/不可点击/不可跳转
     - 开启后未到活动时间、活动进行中、活动结束后等阶段态入口表现
   - 对"外观/展示型功能"，不能只写"若干典型场景"，必须先列出**完整展示场景清单**，再逐场景覆盖；清单至少包含：
     - 功能自身界面
     - 玩家信息相关界面
     - 主界面头像/入口
     - 排行榜/战斗前后/组队/好友/聊天/门派/副本/竞技场等所有会消费该外观产出物的界面
   - 对存在 UI截图说明 / 交互稿 / Excel 图片标注的输入，不能只提炼主流程，必须把截图里可验证的细节转成测试点，例如：
     - 按钮位置、文案、显隐、禁用态
     - 列表布局、行列数、滚动方式、角标、描边、底色
     - 弹窗内容、页签切换、提示icon、说明小窗
     - 倒计时格式、状态文案颜色、资源表现、点击热区
4. 再覆盖数据持久化，按三维度拆分：重启服务器、更换账号（含数据残留）、账号上下线，每个维度逐项验证各数据字段
5. 再覆盖红点系统，按触发源分类：每类覆盖显示条件、消除条件、部分消除、存库持久化
6. 再覆盖弱网、中断、断线重连、多端并发等稳定性场景
7. 再覆盖协议测试，每个写操作接口（解锁、使用、卸下等）需验证非法状态调用、重复调用、参数篡改
8. 最后补充低频或极端边缘场景、视觉与体验类问题

#### 2.4.1 外观/界面型系统专项补充规则

当功能属于头像框、称号、气泡、皮肤、时装、主界面挂件、称号牌、聊天气泡等**外观/界面型系统**或包含**玩家信息展示**时，在通用生成顺序外，必须额外执行以下检查：

1. **入口解锁条件穷举**：先确认是否存在 function_level、activity_time、系统页签解锁、玩家信息页页签解锁、按钮显隐条件等入口门槛；若存在，必须生成开启/关闭对照用例。
2. **展示场景穷举**：先从 spec / Excel / UI稿 / 投放表里列出所有展示入口，形成“展示场景清单”；未形成清单前，不得开始写显示一致性用例。
3. **界面细节穷举**：先按“公共区域 / 当前选中区 / 列表区 / 按钮区 / 弹窗区 / 红点区 / 资源表现区”拆界面，再逐块提炼测试点。
4. **文案与样式成对验证**：凡是状态文案（如已解锁/未解锁/已过期/当前使用），必须同时验证文案内容、颜色样式、图标角标、是否可点击。
5. **截图说明强制落例**：Excel 或 spec 中凡出现“UI截图说明”，必须至少转译出 1 条界面细节用例，不能只作为背景描述略过。
6. **通用玩家信息交互约束**：所有出现“玩家实体”的区域（排行榜、对阵表、聊天、观战等），必须强制验证：**名字的颜色显示（如区分仙阶/阵营专属颜色）**，以及**头像与名字点击能否正常触发玩家详情TIPS界面**。

#### 2.4.2 竞技/赛事/周期性活动专项补充规则

当功能属于阵营竞技、排位赛、巅峰赛、跨服战、门派战等**竞技/赛事/周期性活动**时，必须额外执行以下检查并生成专规用例：

1. **周期与赛季循环逻辑**：必须明确覆盖：活动未开启（周期外）、报名期、预赛期、淘汰赛期、竞猜期、休赛期、下个赛季重置全流程。
2. **参赛人数不足与边界**：必须覆盖：总人数不足（无法凑齐最小比赛规模/跳过某阶段）、分组奇数导致单轮轮空、连续多轮轮空、淘汰赛末端轮空直接晋级等极端情况。
3. **多阵营/多组别异步等待**：当涉及多阵营/多组独立竞赛时，必须覆盖：**某一阵营参赛人数过少导致提前结束预赛，但需处于“界面等待”状态直到其他阵营预赛也结束**的同步/异步时序问题。
4. **实时等待与恢复能力**：赛程阶段、观战界面、跨组匹配期间，必须强制验证：“界面倒计时/状态实时刷新”、“切换后台恢复”、“断网/重连后状态恢复”。
5. **精确的奖励发放节点**：必须按阶段（单局结束、淘汰出局、晋级、决赛完成、竞猜结算、活动整体结束）明确列出产出和奖励发放到账的具体节点。
6. **跨服与分组通用用例**：任何涉及跨服/分组的功能，必须增加以下三个维度的通用用例：
   - ① **按服务器分组规则**：多服混排或指定服互通是否符合设计。
   - ② **开服时间前置门槛**：本服满足“开服时间（如第4天7:15之后）”才可参与分组的约束。
   - ③ **分组结果的确认与持久化**：确认是否分组完成以及合服/滚服对其影响。
7. **多阶段观战界面显示**：赛事在不同阶段（准备中、战斗中、结算后）针对“观战界面”需产生不同的信息显示与下注交互，需独立划分测试分类。

## 3. 输出格式

### 3.1 默认输出策略

- 默认输出测试用例，不输出长篇分析过程
- 若调用链路要求接口消费，默认输出JSON
- 若目标明确为XMind导入，输出Markdown脑图格式
- 若专项Skill明确要求某种格式，则以专项Skill为准

### 3.2 JSON格式（接口生成）

```json
[
  {
    "id": "TC-{taskId}-001",
    "name": "用例名称",
    "priority": "P0",
    "precondition": "前置条件（无则为空字符串）",
    "steps": "1. 第一步操作\n2. 第二步操作",
    "expected": "1. 第一个预期结果\n2. 第二个预期结果"
  }
]
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 固定格式：TC-{taskId}-NNN |
| name | string | 是 | 简洁描述验证点 |
| priority | string | 是 | 仅允许P0/P1/P2 |
| precondition | string | 否 | 无前置时为空字符串 |
| steps | string | 是 | 多步骤用\n分隔 |
| expected | string | 是 | 与步骤逐项对应 |

兼容说明：若下游仍使用level字段，可兼容输出level，但语义与priority保持一致。

### 3.3 Markdown格式（XMind导入）

```markdown
# 功能名称

## 模块名

### 子分类名

- TC-001 用例名称
    - 【前置】前置条件
      【步骤】1. 第一步操作 2. 第二步操作
        - 【预期】1. 第一个预期结果 2. 第二个预期结果
            - 【优先级】P0
```

### 3.4 XMind输出规则

- `#` 表示中心主题
- `##` 表示功能模块
- `###` 表示逻辑子分类
- `- TC-XXX` 表示测试点
- 【前置】与【步骤】合并为一个节点
- 【预期】作为步骤子节点
- 【优先级】作为预期子节点
- 若存在待确认项，应在文档末尾附「附录：待确认事项」

## 4. 生成约束

### 4.1 强制规则

- 测试步骤必须给出明确操作和测试数据
- 预期结果必须可见、可量化、可确认
- 用例编号固定为TC-{taskId}-NNN
- 前置条件无则为空字符串，不写"无"
- 不因资料缺失而停止生成，优先补主流程、边界、状态同步、回归影响
- 若信息冲突，优先保留明确需求，并把不一致处转化为测试关注点或待确认项
- 任何存在入口门槛/功能开关/活动时间门槛的功能，必须同时生成“满足条件→入口显示/可进入”与“不满足条件→入口隐藏/不可进入”两类用例
- 任何外观/界面型系统，必须先产出“展示场景清单”再写显示一致性用例；若无法穷尽，必须在附录列出未覆盖场景
- 任何带 UI截图说明、交互图、Excel图片标注的输入，必须把其中的可验证细节转成测试点，禁止只保留主流程逻辑
- 任何列表/页签/弹窗型界面，必须覆盖布局、层级、显隐、文案、颜色、按钮态、点击热区等至少一轮界面细节验证
- 任何带有“玩家列表/排行榜/对阵表”的UI，强制生成“玩家名字/头像点击出详情Tips”与“名字附带阵营/仙阶对应颜色”的两条通用用例
- 任何赛季/周期性赛事系统，必须穷举“极少人数无法开赛”、“阵营人数不均导致某组提前结束并处于等待同步状态”、“淘汰赛单轮或多轮轮空”边界用例，不得只生成满人理想流程
- 任何跨服赛事系统，必须包含“开服时间门槛”、“按服务器分组规则”、“分组结果持久化”三个维度的确认用例

### 4.2 禁止事项

- 禁止写"正常""成功""无异常"等模糊结果
- 禁止写"测试XX功能"这类非操作性步骤
- 禁止一条用例覆盖多个无关验证点
- 禁止使用不可构造的前置条件
- 禁止遗漏关键测试数据
- 禁止忽略奖励发放、消耗扣除、状态恢复、跨天刷新等高风险点

## 5. 与openspec文档的衔接

当输入来自openspec结构化文档时，按以下方式消费：

| 文档 | 作用 | 用例生成使用方式 |
|------|------|------------------|
| spec.md | 定义Requirement/Scenario | 提取主流程、分支流程、状态变化、验收语义 |
| design.md | 补技术实现、边界和依赖 | 补充状态同步、数据持久化、系统集成、异常恢复 |
| proposal.md | 补业务背景、约束、验收标准 | 补充优先级、回归重点、性能关注、兼容范围 |
| tasks.md | 补实现清单和改动落点 | 补充模块级回归、影响面和联动验证 |

推荐转换顺序：

1. 从spec.md提取必须验证的Requirement和Scenario
2. 从design.md提取状态流转、依赖系统、数据结构、异常处理
3. 从proposal.md提取业务目标、约束条件、验收标准
4. 从tasks.md反推需要补充的回归路径和实现落点
5. 从配置表、UI稿、缺陷记录中补齐精确数值、展示规则和历史回归点

当输入为Excel策划文档时：

- 直接解析Excel内容，不需要查找openspec目录下的md文件
- Excel中的图片/UI截图需要分析解读，结合文本规则解析为测试用例
- 优先提取配置表结构（字段定义、数值范围、状态枚举）作为测试数据源

## 8. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：AI只生成正常流程用例**
- 现象：生成20条用例全是"正常xxx"，无边界/异常/并发
- 原因：Prompt未明确分类覆盖要求
- 解决：按2.4节生成思路顺序强制要求覆盖

**坑2：预期结果全是"功能正常"**
- 现象：预期结果写"购买成功"而非"钻石减少100，道具ID:10001数量+1"
- 原因：AI倾向于生成笼统描述
- 解决：Prompt中明确禁止模糊表述，要求具体数值/状态

**坑3：前置条件无法构造**
- 现象：用例要求"玩家等级60且拥有传说装备"，但无对应GM指令
- 原因：AI不了解实际可用的GM指令
- 解决：输入中必须提供GM指令清单

**坑4：遗漏跨系统影响**
- 现象：经济系统用例只验证货币变动，未验证是否触发成就/任务
- 原因：AI按单功能思维生成
- 解决：引用R23规则，要求标注跨系统影响

**坑5：优先级标注不准**
- 现象：把奖励发放用例标为P2，把视觉细节标为P0
- 原因：AI未理解优先级维度
- 解决：严格按2.1节五维度判定，资产风险默认P0

**坑6：涉及属性/战力变动的功能遗漏战斗公式验证**
- 现象：装扮系统有属性加成（attr_value），解锁后会增加战力buff，但AI和手写用例都只验证了"属性面板数值变化"和"加成总览弹窗更新"，完全没有验证这个buff是否真正参与了战斗伤害计算。即：属性面板显示+5%攻击，但实际战斗中伤害公式是否正确吃到了这个加成？
- 原因：AI和人工都习惯在"功能自身"范围内验证数值正确性（面板显示对了就算过），忽略了属性加成的最终消费方是战斗系统。属性面板正确 ≠ 战斗公式正确，中间可能存在buff类型未注册、加成层级计算顺序错误、百分比与固定值叠加规则不一致等问题
- 解决：任何涉及属性/战力/buff变动的功能，必须增加"战斗公式验证"环节：①解锁/激活前后分别进入战斗，对比同一目标的实际伤害值 ②用伤害公式手算预期值，与实际值比对（引用R29规则） ③验证多个属性加成叠加时的计算顺序（百分比加成 vs 固定值加成，引用R24规则） ④验证buff移除后（过期/卸下）伤害值正确回退

**坑7：装扮/外观类系统遗漏多场景显示一致性验证**
- 现象：AI生成的用例只验证了装扮的解锁/使用/过期逻辑，完全没有覆盖装扮在各功能界面（排行榜、竞技场、副本、好友、聊天、门派、灵脉等）的显示一致性。手写用例中有75条逐场景验证，AI生成0条。
- 原因：AI按"功能自身逻辑"思维生成，缺少"该功能的产出物在其他系统中如何展示"的横向思维
- 解决：对于装扮/头像/称号/气泡/皮肤等外观类系统，必须增加"显示场景覆盖"环节——列举所有会展示该外观的功能界面，逐一验证显示正确性。建议在生成思路中增加一步："识别功能产出物的所有展示入口，逐场景验证"

**坑8：数据存库验证只写笼统"重登一致"，缺少三维度拆分**
- 现象：AI只生成了"重登后状态一致""服务端重启后恢复"两条笼统用例，手写用例按"重启服务器/更换账号/账号上下线"三个维度分别验证每项数据（解锁进度、解锁状态、使用状态），共30+条
- 原因：AI把数据持久化当作一个整体验证点，没有按触发场景和数据项拆分
- 解决：数据存库类用例必须按三个维度拆分：①重启服务器 ②更换账号（含数据残留检查） ③账号上下线。每个维度下需逐项验证各独立数据字段的存库正确性

**坑9：红点系统只验证"有红点/消红点"，缺少触发源分类和存库**
- 现象：AI只生成了2条红点用例（全链路显示+解锁消除），手写用例按触发源分为"待解锁装扮红点""获得新装扮红点""限时过期红点"三类，每类分别验证显示/消除/部分消除，加上红点存库共10条
- 原因：AI把红点当作单一功能点，没有按触发源拆分
- 解决：红点类用例必须按触发源分类生成，每类需覆盖：①红点出现条件 ②红点消除条件 ③部分消除（多个红点源只处理一个时其余保留） ④红点存库持久化（重启/换号/上下线后不重复出现）

**坑10：功能入口的配置表开关条件未覆盖**
- 现象：AI直接从"进入装扮界面"开始写用例，没有验证function_level等配置表控制的功能开启/关闭条件
- 原因：AI默认功能已开启，忽略了"功能入口本身就是测试点"
- 解决：任何有开启条件的功能，必须先验证"满足条件→入口显示""不满足条件→入口隐藏"，这是P0级用例

**坑11：条件解锁类型覆盖不全**
- 现象：AI覆盖了累计登录、师父快跑、神武副本带人三种解锁条件，遗漏了"神将解锁"条件
- 原因：AI从常见模式推断解锁类型，没有穷举配置表中所有解锁条件类型
- 解决：条件解锁类用例必须先从配置表中穷举所有解锁条件类型，逐一生成对应的达成/未达成用例，不能靠推断

**坑12：协议测试只覆盖解锁侧，遗漏使用侧**
- 现象：AI生成了"协议篡改解锁"和"协议重放解锁"，但没有生成"协议使用未解锁装扮""协议使用已穿戴装扮""协议使用已过期装扮"
- 原因：AI按功能模块生成协议测试，只在解锁模块下生成了协议用例，使用模块下遗漏
- 解决：协议测试必须覆盖每个写操作接口（解锁、使用、卸下），每个接口需验证：非法状态调用、重复调用、参数篡改

**坑13：入口功能只写进入路径，遗漏入口解锁条件和关闭态**
- 现象：AI从"点击入口进入功能"开始写主流程，却没有验证功能未解锁、未到开放时间、活动关闭、页签未开启等情况下入口是否隐藏/禁用。
- 原因：AI把入口当成固定前提，没有把入口本身视为一个独立状态机。
- 解决：任何有入口的系统，必须先判断是否存在入口解锁条件、活动开启时间、页签显示条件、按钮显隐条件；若存在，必须生成开启态和关闭态两组对照用例。

**坑14：显示场景只写示例，不先穷举完整场景清单**
- 现象：AI只写了主界面、聊天、排行榜等几个典型场景，就停止生成显示一致性用例，遗漏好友、门派、副本、竞技场、组队、结算、详情页等消费外观的界面。
- 原因：AI按"举例覆盖"而不是"场景穷举"工作，缺少先列清单再写用例的步骤。
- 解决：对所有外观/展示型系统，必须先单独产出"展示场景清单"；清单完成后逐项映射成用例。若 spec 未写全，需结合投放表、UI稿、历史经验补全，并把无法确认的场景写入附录。

**坑15：UI截图说明被当作背景材料，没有转成界面细节用例**
- 现象：spec 中已经写明了按钮位置、列表布局、弹窗结构、状态文案和视觉表现，但AI生成时只保留了解锁/使用/过期等主逻辑，遗漏了标题、按钮位置、icon置灰、金色描边、倒计时格式、提示icon等界面细节。
- 原因：AI把 UI截图说明当作参考描述，没有建立“截图说明 = 可验证需求”的约束。
- 解决：凡是输入资料包含 UI截图说明、Excel图片标注、交互图，必须先拆成“可点击区 / 显示区 / 状态区 / 反馈区 / 资源区”五类细节，再逐类生成用例；至少保证每块界面都有可执行测试点。

**坑16：周期性赛事忽略赛季循环与极端人数分配**
- 现象：AI按理想的"8进4"流程生成，遗漏了休赛期表现、单阵营不满人数、各组进度不一致导致的异步等待，以及多轮轮空情况。
- 原因：AI默认赛事是满人且同步进行的，没有考虑服务器真实的报名生态和多进程时序。
- 解决：任何赛事玩法，必须穷举奇数、极少人数、阵营人数不均导致进度差异的边界，并增加"界面等待实时刷新/断线重连"用例。

**坑17：跨服分组与通用UI规范被遗漏**
- 现象：赛事没有验证服务器分组逻辑（如开服时间前置条件）；UI上玩家名字没有带上阵营专属颜色，点击头像也没有弹出详情Tips。
- 原因：AI把目光局限在玩法逻辑本身，忽略了跨服前置条件与全游戏通用的UI交互标准。
- 解决：跨服玩法必须增加"按服务器分组"、"开服天数限制"、"确认分组完成"前置用例。所有出现玩家实体的列表，强制增加"名字带阵营/仙阶颜色"和"头像名字可点出详情Tips"的通用UI检查点。
