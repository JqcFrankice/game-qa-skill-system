# 测试用例生成规范

> 所属板块: 流程与工具能力 | 编号: T08 | 最后更新: 2026-03-30

---

## 1. 概述

测试用例生成规范定义了AI（LLM）生成游戏测试用例时的输入格式、输出模板、质量校验规则和通用检查清单。本Skill是T04（测试设计）和T07（AI辅助测试）的桥梁——T04定义"怎么设计用例"，T07定义"怎么让AI执行"，本Skill定义"怎么让AI生成高质量用例"。

核心原则：**AI生成 + 人工校验 + 规则兜底**。AI负责批量生成，人工负责审查，29条通用规则作为硬性检查清单。

---

## 2. 知识体系

### 2.1 AI生成用例的工作流

```
输入（项目特性 + 功能描述 + Skill文档）
  ↓
AI生成（按输出模板 + 遵循生成规则）
  ↓
自动校验（29条通用规则逐条检查）
  ↓
人工审查（补充业务判断、边界场景）
  ↓
入库执行
```

### 2.2 输入规范

AI生成用例前，必须提供以下上下文：

| 输入项 | 必填 | 来源 | 说明 |
|--------|------|------|------|
| 项目特性档案 | ✅ | G00_项目特性 | 项目类型/引擎/网络模型/付费模型 |
| 目标功能描述 | ✅ | 策划文档/需求文档 | 功能的业务逻辑、状态流转、数值公式 |
| 对应Skill文档 | ✅ | F01-F12/S01-S09等 | 该功能领域的知识体系和已有用例框架 |
| GM指令清单 | ✅ | G00_项目特性 | 可用的GM指令及参数范围 |
| 数值配置表 | 推荐 | 策划数值表 | 公式/概率/阈值/上限等具体数值 |
| 历史Bug清单 | 推荐 | 缺陷管理系统 | 该功能历史高频Bug，防止遗漏回归 |
| 踩坑经验库 | 推荐 | 各Skill文档第7章 | 已知坑点，AI需针对性生成覆盖用例 |

### 2.3 输出模板

每条AI生成的用例必须包含以下字段：

```yaml
- case_id: "{Skill编号}-{分类缩写}-{序号}"   # 如 F04-C03
  title: ""                                    # 一句话描述场景
  precondition: ""                             # 前置条件（含GM指令）
  steps:                                       # 操作步骤（编号列表）
    - "1. xxx"
    - "2. xxx"
  expected: ""                                 # 预期结果（可量化）
  priority: "P0/P1/P2/P3"                       # 优先级（见2.4节分级规范）
  priority_basis: ""                           # 分级依据（路径重要度+用户范围+操作频率）
  category: ""                                 # 分类（正常流程/边界/异常/并发/安全）
  related_rules: []                            # 命中的通用规则编号（如 [R01, R04]）
  gm_commands: []                              # 用到的GM指令
  verify_method: ""                            # 验证方式（UI观察/协议抓包/数据库查询/日志）
```

### 2.4 测试用例分级规范

测试用例等级分为P0、P1、P2、P3四个等级，基于三个维度综合判定：

#### 2.4.1 等级划分维度

**维度一：功能路径的重要度**

| 路径类型 | 定义 | 示例 |
|---------|------|------|
| 主干功能路径 | 用户为完成产品核心目标所必须连续执行的操作序列，处于整个使用流程的中心干道上 | 登录→进入主界面→开始战斗→结算 |
| 分支功能路径 | 从主干路径中的某个节点分岔出去的可选操作路径，不影响主干流程的继续推进 | 查看排行榜、修改阵容、查看成就 |
| 边缘功能路径 | 远离主干流程、通常需主动深度导航或多层跳转才能访问的路径 | 修改头像框、查看版本号、分享截图 |

> 无论该功能是否"重要"，完成路径越靠近用户完成核心目标的主干流程，其重要度越高，对应测试用例优先级应提升。

**维度二：影响的用户范围**

| 范围 | 定义 | 示例 |
|------|------|------|
| 全量用户 | 问题或功能覆盖所有用户 | 注册、登录、新手引导 |
| 群组用户 | 覆盖部分用户群体 | 付费用户、某渠道用户、未满18岁用户 |
| 个例用户 | 仅覆盖特定特征的用户 | 背包数据溢出的用户、利用技术手段作弊的用户 |

> 影响用户基数越大，测试用例应越优先覆盖，尤其需保障高覆盖率和稳定性验证。

**维度三：用户的操作频率**

| 频率 | 定义 | 示例 |
|------|------|------|
| 高频操作 | 每次游戏时多次使用的功能 | 签到、进入对局、领取奖励、切后台 |
| 中频操作 | 阶段性使用的功能 | 添加好友、联机邀请 |
| 低频操作 | 较少使用的功能或触发的操作 | 修改昵称、分享邀请 |

> 操作频率越高，功能稳定性对整体体验影响越大，对应测试用例应优先执行并加强回归覆盖。

#### 2.4.2 等级定义与标准

| 用例等级 | 测试目标 | 覆盖范围 |
|---------|---------|---------|
| **P0** | 验证系统基本流程、核心逻辑是否正常。此级别用例的失败会直接影响系统可用性和用户核心体验。 | 业务主流程的正常运行，直接影响系统可用性的主干路径。包括：功能入口、功能解锁、功能运行、功能关闭、核心功能循环、奖励发放等。 |
| **P1** | 验证整个功能逻辑的完备性。此级别用例的失败会导致部分功能/用户数据缺失，或出现较影响用户体验的问题。 | 业务分支流程的正常运行，影响系统功能完整性的路径。包括：分支功能、边界条件、高频使用场景、重要信息显示及交互、数据存盘/读取等。 |
| **P2** | 验证网络、中断、协议相关的正确性。此级别用例相关问题易发性较高，且可能造成较严重的影响。 | 出现概率高、影响较严重且范围广泛的场景。包括：弱网、中断、断线重连、协议验证。 |
| **P3** | 验证极端场景、主观体验等非核心功能的正确性。此级别用例相关问题影响范围小，且易发性较低。 | 发生概率低、影响范围小、低频操作相关的场景。包括：耦合情况、用户异常操作、体验性内容等。 |

#### 2.4.3 分级示例：抽卡系统

**功能流程：** 玩家发起抽卡 → 服务器校验(货币/权限) → 概率计算(概率表/保底) → 发放奖励(入包/重复转碎片) → 记录日志 → 客户端展示(动画/弹窗)

| 等级 | 功能点 | 测试点 |
|------|--------|--------|
| **P0** | 抽卡玩法入口 | 入口开关、玩法参与条件 |
| | 单次抽卡&十连抽 | 抽卡操作、货币扣减、道具发放 |
| | 概率计算 | 概率计算正确性、随机算法、保底机制触发 |
| **P1** | 单次抽卡&十连抽 | 重复奖励处理、抽卡动画展示、大奖跑马灯 |
| | 分支功能 | 抽卡记录查询、概率展示、奖池内容预览 |
| **P2** | 中断相关 | 抽卡过程中各种中断（来电/切后台/杀进程） |
| | 弱网相关 | 弱网抽卡、网络切换 |
| | 数据相关 | 奖励/保底次数数据存盘与读取 |
| **P3** | 单次抽卡&十连抽 | 立绘效果、界面动效 |
| | 协议相关 | 抽卡协议字段验证 |
| | 异常操作 | 快速连点抽卡、背包满时抽卡 |

### 2.5 用例分类标准

AI生成用例时，必须按以下分类覆盖，不得只生成正常流程：

| 分类 | 占比要求 | 说明 |
|------|---------|------|
| 正常流程 | 30% | 主干功能的正向验证 |
| 边界值 | 20% | 上限/下限/零值/满仓/过期 |
| 异常场景 | 25% | 断线/崩溃/超时/服务端重启 |
| 并发/竞态 | 15% | 快速双击/多端操作/结算瞬间 |
| 安全 | 10% | 协议篡改/重放/负数注入 |

## 3. 通用规则检查清单（29条）

以下规则是AI生成用例后的硬性校验清单。每条用例生成后，AI必须自检是否遗漏了相关规则的覆盖。

### 3.1 数值与资产安全（R01-R04）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R01 | 所有数值累加必须考虑 **int32溢出**，货币/经验/属性/MMR需上限保护 | GM设极值后触发增加操作 |
| R02 | 货币计算严禁浮点数，必须用 **整数（分为单位）** | 折扣/兑换场景验证精度 |
| R03 | 所有资产变动服务端必须做 **幂等校验**，防止重复发放 | 重复发送领取/提交请求 |
| R04 | 任何消耗操作必须验证 **并发安全**，不得双花或负数 | 快速双击/多端同时操作 |

### 3.2 协议与接口（R05-R07）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R05 | 涉及资产变动的功能必须 **抓包验证协议** | 抓包对比请求/响应字段 |
| R06 | 关键操作必须防 **重放攻击** | 截获合法请求重复发送 |
| R07 | 客户端参数服务端必须做 **合法性校验** | 传负数/超范围/篡改值 |

### 3.3 时间相关（R08-R10）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R08 | 涉及时间的功能必须考虑 **跨天/跨周/跨赛季** | GM修改时间到零点瞬间操作 |
| R09 | 时间判定必须以 **服务端时间为准** | 客户端改时间后验证业务 |
| R10 | 多时区功能必须验证 **时区一致性** | 不同时区账号对比 |

### 3.4 断线与异常恢复（R11-R13）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R11 | 关键操作节点必须测 **断线重连**，操作原子性 | 操作中断网→重连 |
| R12 | 崩溃/重启后数据必须 **以服务端持久化为准恢复** | 强杀客户端→重登 |
| R13 | 重连快照必须完整，含 **Buff/进度/战斗/匹配状态** | 断线重连后逐项检查 |

### 3.5 边界与容量（R14-R16）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R14 | 有上限的功能必须验证 **满仓/满额兜底策略** | GM填满后触发获取 |
| R15 | 堆叠数据超上限时溢出部分 **不得丢失** | 堆叠999再获取 |
| R16 | 过期数据必须验证 **过期时机准确性**和使用时二次校验 | 过期前后分别操作 |

### 3.6 状态机与流转（R17-R19）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R17 | 有状态流转的系统必须验证 **非法状态转换被拒绝** | 协议直接发非法状态 |
| R18 | 前置条件功能必须验证 **GM跳过是否触发完整事件链** | GM完成前置→检查后续 |
| R19 | UI层级功能必须验证 **遮挡和点击穿透** | 引导/弹窗同时触发 |

### 3.7 并发与竞态（R20-R22）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R20 | 结算瞬间必须验证 **与进行中操作的竞态** | 结算时刻发起操作 |
| R21 | 排行榜/结算必须 **冻结数据写入** | 结算中修改积分 |
| R22 | 所有"领取"操作客户端加 **点击CD**，服务端加幂等 | 快速连点领取按钮 |

### 3.8 跨系统联动（R23-R25）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R23 | 资产变动必须验证 **跨系统影响** | GM加金币→检查成就/任务 |
| R24 | 多源属性叠加必须明确 **百分比与固定值计算顺序** | 多Buff叠加→对比公式 |
| R25 | 穿戴/激活/移除必须验证 **属性正确加减** | 穿→脱→穿循环检查 |

### 3.9 内容安全（R26-R27）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R26 | 用户输入文本必须过 **敏感词过滤**（含变体） | 谐音/零宽字符/符号混淆 |
| R27 | 支付功能必须验证 **签名校验和金额防篡改** | 抓包修改金额/签名 |

### 3.10 概率与公式（R28-R29）

| 编号 | 规则 | 校验方法 |
|------|------|---------|
| R28 | 概率功能必须 **批量统计验证**，偏差≤±3% | 1000次操作统计 |
| R29 | 伤害/属性公式必须 **逐条比对策划数值表** | Excel预算值做基准 |

## 4. AI生成Prompt模板

### 4.1 标准Prompt结构

```markdown
# 角色
你是一个资深游戏QA，负责为{项目名}的{功能名}生成测试用例。

# 输入上下文
## 项目特性
{粘贴G00项目特性档案的关键字段}

## 功能描述
{粘贴策划文档/需求描述}

## 数值配置
{粘贴相关数值表/公式}

## 可用GM指令
{粘贴相关GM指令清单}

## 历史踩坑
{粘贴该功能相关的踩坑经验}

# 生成要求
1. 按以下YAML格式输出每条用例
2. 分类覆盖：正常流程30%、边界值20%、异常场景25%、并发竞态15%、安全10%
3. 每条用例必须标注命中的通用规则编号（R01-R29）
4. 前置条件必须使用GM指令构造，不得依赖手动操作
5. 预期结果必须可量化验证，不得使用"正常"、"没问题"等模糊描述
6. 用例分级必须严格遵循2.4节的P0-P3分级规范，每条用例标注分级依据
7. P0=主干流程+核心逻辑，P1=分支功能+边界+高频场景，P2=弱网/中断/协议，P3=极端场景+体验性内容

# 输出格式
{粘贴2.3节的YAML模板}

# 禁止
- 不得生成无法通过GM指令构造前置条件的用例
- 不得遗漏并发和安全分类
- 预期结果不得使用模糊表述
- 不得只生成正常流程用例
```

## 5. 质量校验规则

### 5.1 AI自检清单（生成后自动执行）

| 检查项 | 通过标准 | 不通过处理 |
|--------|---------|-----------|
| 分类覆盖率 | 5个分类均有用例 | 补充缺失分类 |
| 规则覆盖率 | 该功能相关的R规则均被至少1条用例命中 | 补充遗漏规则的用例 |
| 前置条件可执行 | 所有前置条件使用GM指令或明确操作步骤 | 改写为GM指令构造 |
| 预期结果可量化 | 不含"正常"/"没问题"/"应该"等模糊词 | 改写为具体数值/状态 |
| 分级规范合规 | P0-P3分级符合2.4节三维度标准，P0仅含主干路径 | 按分级规范重新标注 |
| 步骤可复现 | 步骤编号连续，无跳跃，无歧义 | 改写步骤 |

### 5.2 人工审查要点

| 审查项 | 说明 |
|--------|------|
| 业务逻辑正确性 | AI可能误解策划意图，需人工确认预期结果 |
| 边界值完整性 | AI可能遗漏特定业务的边界（如VIP等级、赛季边界） |
| 踩坑经验覆盖 | 对照踩坑经验库，确认历史坑点均有对应用例 |
| GM指令可用性 | 确认用例中的GM指令在当前环境实际可用 |
| 分级合理性 | 按功能路径重要度+用户范围+操作频率三维度校准P0-P3等级 |

### 5.3 用例质量评分

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 规则覆盖率 | 30% | 命中R规则数 / 该功能应命中R规则总数 |
| 分类覆盖率 | 20% | 实际覆盖分类数 / 5 |
| 可执行性 | 20% | 前置条件+步骤可直接执行的用例占比 |
| 预期明确性 | 15% | 预期结果可量化验证的用例占比 |
| 边界覆盖 | 15% | 边界值+异常用例占比是否达标(≥45%) |

合格线：总分 ≥ 80分。低于80分需补充后重新评分。

| LLM（GPT/Claude/Copilot） | 用例批量生成 | 按Prompt模板输入 |
| YAML/JSON解析器 | 用例格式校验 | 确保输出格式合规 |
| 规则校验脚本 | 自动检查R规则覆盖率 | 可自研或用LLM二次校验 |

## 7. 执行流程

### 7.1 准备阶段
- 确认项目特性档案(G00)已填写完整
- 收集目标功能的策划文档、数值表、GM指令
- 选择对应Skill文档作为领域知识输入
- 收集该功能的历史Bug和踩坑经验

### 7.2 生成阶段
- 按4.1节Prompt模板组装输入
- 按4.2节补充领域专属指令
- AI生成用例，输出YAML格式

### 7.3 校验阶段
- AI自检：按5.1节清单逐项检查
- 规则覆盖：确认相关R规则均被命中
- 人工审查：按5.2节要点逐条审查
- 质量评分：按5.3节评分，低于80分需补充

### 7.4 入库阶段
- 通过审查的用例导入用例管理系统
- 标注用例来源（AI生成+人工审查）
- 关联对应Skill文档和R规则编号

## 8. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：AI只生成正常流程用例**
- 现象：生成20条用例全是"正常xxx"，无边界/异常/并发
- 原因：Prompt未明确分类覆盖要求
- 解决：Prompt中强制要求分类占比，自检时校验

**坑2：预期结果全是"功能正常"**
- 现象：预期结果写"购买成功"而非"钻石减少100，道具ID:10001数量+1"
- 原因：AI倾向于生成笼统描述
- 解决：Prompt中明确禁止模糊表述，要求具体数值/状态

**坑3：前置条件无法构造**
- 现象：用例要求"玩家等级60且拥有传说装备"，但无对应GM指令
- 原因：AI不了解实际可用的GM指令
- 解决：输入中必须提供GM指令清单，Prompt中要求前置条件只用GM指令

**坑4：遗漏跨系统影响**
- 现象：经济系统用例只验证货币变动，未验证是否触发成就/任务
- 原因：AI按单功能思维生成，缺乏跨系统视角
- 解决：Prompt中引用R23规则，要求每条资产变动用例标注跨系统影响

**坑5：概率类用例缺少统计方法**
- 现象：用例写"验证强化成功率为50%"，步骤只执行1次
- 原因：AI不理解概率验证需要大样本
- 解决：Prompt中引用R28规则，要求概率用例步骤包含"执行N次并统计"
