# 测试设计

> 所属板块: 流程与工具能力 | 编号: T04 | 最后更新: 2026-03-29

---

## 1. 概述

测试设计是将测试需求转化为可执行测试用例的核心能力，直接决定测试覆盖的广度和深度。掌握等价类划分、边界值分析、正交试验、场景法、探索性测试、风险驱动等方法，并能根据项目特点灵活组合运用，是QA专业能力的重要体现。游戏测试设计需特别关注数值平衡、随机性验证、多人交互等特殊场景。

## 2. 知识体系

### 2.1 等价类划分

将输入域划分为若干等价类，从每个等价类中选取代表值进行测试。

- 有效等价类：符合规格说明的合法输入
- 无效等价类：不符合规格说明的非法输入

**游戏应用示例 - 角色命名：**

| 等价类 | 类型 | 代表值 | 预期 |
|--------|------|--------|------|
| 2-12位中英文 | 有效 | "玩家Test" | 创建成功 |
| 空字符串 | 无效 | "" | 提示名称不能为空 |
| 超过12位 | 无效 | "这是一个超长名字测试" | 提示名称过长 |
| 含特殊字符 | 无效 | "玩家@#$" | 提示含非法字符 |
| 含敏感词 | 无效 | (敏感词) | 提示名称不可用 |
| 纯空格 | 无效 | "   " | 提示名称不能为空 |

### 2.2 边界值分析

在等价类边界处取值测试，因为边界处最容易出错。

**取值策略：** 边界值、边界值±1(上点、离点、内点)

**游戏应用示例 - 背包容量(上限100)：**

| 边界 | 测试值 | 预期 |
|------|--------|------|
| 下界-1 | 0个物品时使用 | 提示背包为空 |
| 下界 | 1个物品 | 正常操作 |
| 上界-1 | 99个物品时获取 | 获取成功，剩余1格 |
| 上界 | 100个物品时获取 | 提示背包已满 |
| 上界+1 | 尝试超过100 | 溢出物品进入邮箱/拒绝 |

### 2.3 正交试验法

当多个因素组合测试时，使用正交表选取代表性组合，在保证覆盖度的前提下减少用例数。

**适用场景：** 多条件筛选、多参数配置、多因素组合

**游戏应用示例 - 装备强化：**
- 因素：装备品质(白/绿/蓝/紫/橙)、强化等级(1-15)、使用保护符(是/否)、角色等级(1-60)
- 全组合：5×15×2×60 = 9000种
- 正交表：L25(5^6) 可缩减至25-50条用例覆盖主要组合

### 2.4 场景法

基于用户使用场景设计测试，关注业务流程的完整性。

**基本流(Happy Path)：** 最常见的正常操作路径
**备选流(Alternative Flow)：** 分支路径
**异常流(Exception Flow)：** 错误处理路径

**游戏应用示例 - 组队副本：**
- 基本流：创建队伍→邀请队友→全员就绪→进入副本→通关→结算奖励
- 备选流1：队长中途离线→副队长接管→继续战斗
- 备选流2：队员拒绝邀请→重新邀请/换人
- 异常流1：副本中服务器断连→重连恢复进度
- 异常流2：结算时背包满→奖励发送邮箱

### 2.5 探索性测试

基于测试者的经验、直觉和创造力，同时进行学习、设计和执行。

**Session-Based Test Management (SBTM)：**

| 要素 | 说明 |
|------|------|
| Charter(章程) | 本次探索的目标和范围 |
| Time Box(时间盒) | 通常60-90分钟 |
| Session Sheet(记录表) | 记录发现、问题、疑问 |
| Debrief(汇报) | 会后总结发现与覆盖情况 |

**探索性测试启发式：**
- SFDPOT：Structure/Function/Data/Platform/Operations/Time
- 漫游测试法：地标测试、极端旅行、深夜测试、破坏者测试

### 2.6 风险驱动测试

根据风险评估结果分配测试资源，高风险区域投入更多测试。

**风险评估矩阵：**

| | 影响-低 | 影响-中 | 影响-高 |
|---|--------|--------|--------|
| 概率-高 | 中风险 | 高风险 | 极高风险 |
| 概率-中 | 低风险 | 中风险 | 高风险 |
| 概率-低 | 极低风险 | 低风险 | 中风险 |

**游戏高风险区域识别：**
- 新开发的核心玩法模块
- 涉及货币/交易的功能
- 多人实时交互功能
- 频繁修改的代码模块
- 第三方SDK集成点

### 2.7 用例评审方法

- 同行评审：QA之间互审，关注覆盖完整性
- 开发评审：开发参与，补充技术实现角度的用例
- 策划评审：策划参与，确认业务逻辑理解正确
- 检查清单法：基于历史经验的检查项逐条核对

### 2.8 测试覆盖率度量

| 覆盖率类型 | 定义 | 适用场景 |
|-----------|------|---------|
| 需求覆盖率 | 已覆盖需求点/总需求点 | 功能测试 |
| 用例覆盖率 | 已执行用例/总用例 | 测试进度 |
| 代码覆盖率 | 已覆盖代码行/总代码行 | 单元测试/白盒测试 |
| 场景覆盖率 | 已覆盖场景/总场景 | 集成测试 |
| 风险覆盖率 | 已覆盖风险项/总风险项 | 风险驱动测试 |

## 5. 执行流程

### 5.1 准备阶段
- 研读需求文档/策划案，梳理功能点清单
- 与策划/开发沟通确认疑问点
- 选择合适的测试设计方法(或组合)
- 绘制测试分析脑图，识别测试点
- 评估风险，确定测试重点

### 5.2 执行阶段
- 按选定方法编写测试用例
- 组织用例评审(QA+开发+策划)
- 根据评审意见修改完善用例
- 标注用例优先级，建立执行顺序
- 关联需求与用例的追溯矩阵

### 5.3 分析阶段
- 统计需求覆盖率，识别遗漏
- 分析用例分布(按模块/类型/优先级)
- 评估用例设计质量(有效性/可维护性)
- 对比历史版本用例复用率
- 根据测试执行结果反馈优化用例

### 5.4 报告阶段
- 输出测试设计覆盖率报告
- 汇总用例评审记录与改进项
- 更新用例库(新增/修改/废弃)
- 沉淀测试设计经验到知识库
- 输出探索性测试Session报告

## 6. 数据指标

| 指标 | 定义 | 基线值 | 告警阈值 |
|------|------|--------|---------|
| 需求覆盖率 | 已覆盖需求点/总需求点 | ≥100% | <95% |
| 用例评审通过率 | 一次评审通过用例数/总用例数 | ≥90% | <80% |
| 用例有效率 | 发现Bug的用例数/总执行用例数 | ≥15% | <5% |
| 用例复用率 | 可复用用例数/总用例数 | ≥60% | <40% |
| 探索性测试Bug发现率 | 探索性测试发现Bug数/总Bug数 | ≥20% | <10% |
| 用例设计效率 | 每人日设计用例数 | 20-40条 | <10条 |
| 遗漏缺陷率 | 线上Bug中用例未覆盖的占比 | <10% | >20% |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：用例设计只关注正向流程，忽略异常场景**
- 现象：功能测试通过但上线后频繁出现异常情况的Bug
- 原因：测试设计时只考虑了Happy Path，未充分考虑异常/边界/容错
- 解决：强制要求每个功能点至少30%的异常用例，使用SFDPOT启发式补充

**坑2：正交组合爆炸，用例数量失控**
- 现象：多因素组合导致用例数成千上万，无法全部执行
- 原因：未使用正交表/Pairwise方法优化组合
- 解决：使用PICT工具生成Pairwise组合，结合风险评估裁剪低优先级组合

**坑3：用例描述模糊，不同人执行结果不同**
- 现象：同一条用例A测通过B测失败
- 原因：步骤描述不够精确，预期结果不够明确
- 解决：步骤精确到具体操作和数据，预期结果量化(具体数值/状态/提示文案)

**坑4：探索性测试流于形式，无有效产出**
- 现象：探索性测试Session结束后没有发现有价值的Bug
- 原因：缺少明确的Charter，测试者经验不足，未使用启发式方法
- 解决：制定明确的探索Charter，培训启发式方法，Session后Debrief总结

**坑5：用例维护不及时，大量过期用例**
- 现象：需求变更后用例未同步更新，执行时发现大量用例已失效
- 原因：用例与需求未建立追溯关系，变更时无法识别影响范围
- 解决：建立需求-用例追溯矩阵，需求变更时自动标记关联用例待更新
