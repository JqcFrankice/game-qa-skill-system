# 缺陷管理

> 所属板块: 流程与工具能力 | 编号: T03 | 最后更新: 2026-03-29

---

## 1. 概述

缺陷管理是QA工作的核心流程之一，涵盖Bug的发现、记录、跟踪、修复、验证到关闭的完整生命周期。高效的缺陷管理不仅能加速问题修复，还能通过数据分析驱动质量改进。游戏项目因迭代快、模块耦合度高、多端并行等特点，对缺陷管理的规范性和分析深度要求更高。

## 2. 知识体系

### 2.1 缺陷分类标准

**按模块分类：**

| 分类 | 子类 | 示例 |
|------|------|------|
| 功能缺陷 | 逻辑错误/流程中断/数据异常 | 技能伤害计算错误、任务无法完成 |
| UI缺陷 | 显示异常/适配问题/文本错误 | 按钮遮挡、文字溢出、翻译缺失 |
| 性能缺陷 | 卡顿/内存泄漏/加载慢 | 战斗掉帧、场景加载超时 |
| 崩溃缺陷 | Crash/ANR/闪退 | 空指针崩溃、内存溢出 |
| 兼容性缺陷 | 机型/系统/分辨率适配 | 特定机型花屏、系统版本不兼容 |
| 网络缺陷 | 断线/延迟/同步异常 | 弱网卡死、数据不同步 |
| 安全缺陷 | 协议漏洞/数据篡改/外挂 | 内存修改、协议重放攻击 |

### 2.2 严重程度 vs 优先级

**严重程度(Severity)** - 技术影响维度：

| 等级 | 定义 | 示例 |
|------|------|------|
| S1-致命 | 系统崩溃/数据丢失/核心功能不可用 | 登录崩溃、充值丢单、存档损坏 |
| S2-严重 | 核心功能异常/严重影响体验 | 战斗数值错误、主线任务卡死 |
| S3-一般 | 非核心功能异常/有规避方案 | 成就未触发、排行榜刷新延迟 |
| S4-轻微 | 体验瑕疵/美术细节 | 文字对齐、图标模糊、动画瑕疵 |

**优先级(Priority)** - 业务决策维度：

| 等级 | 定义 | 修复时限 | 决策因素 |
|------|------|---------|---------|
| P0-紧急 | 必须立即修复 | 4小时内 | 影响线上运营/大量用户 |
| P1-高 | 当前迭代必须修复 | 当前Sprint | 影响核心玩法/即将上线功能 |
| P2-中 | 计划修复 | 1-2个Sprint | 非核心功能/有规避方案 |
| P3-低 | 有空再修 | 无明确时限 | 体验优化/低频场景 |

> 严重程度和优先级不一定正相关。例如：S1的崩溃如果只在极罕见机型出现，优先级可能是P2；S4的文字错误如果出现在首页，优先级可能是P1。

### 2.3 DI值(Defect Index)计算

DI值用于量化版本缺陷密度，评估版本质量：

```
DI = (致命Bug数 × 10 + 严重Bug数 × 3 + 一般Bug数 × 1 + 轻微Bug数 × 0.1) / 功能点数
```

| DI值范围 | 质量评估 | 建议 |
|---------|---------|------|
| DI < 1 | 优秀 | 可发布 |
| 1 ≤ DI < 3 | 良好 | 关注严重Bug修复情况 |
| 3 ≤ DI < 5 | 一般 | 需加强测试，修复高优Bug |
| DI ≥ 5 | 较差 | 建议延期发布，集中修复 |

### 2.4 缺陷收敛模型

**Rayleigh模型：** 缺陷发现率随时间呈先升后降的曲线，峰值出现在项目中期。

**收敛判断标准：**
- 每日新增Bug数持续下降且趋于稳定
- 累计Bug修复率 ≥ 95%
- 遗留S1/S2 Bug数 = 0
- DI值达标
- 新增Bug中S1/S2占比 < 5%

**收敛曲线关注点：**
- 如果后期Bug数突然上升 → 可能引入了新功能或回归问题
- 如果Bug数长期不收敛 → 代码质量或测试覆盖有问题
- 如果Bug数过早收敛 → 可能测试不充分

### 2.5 根因分析方法

**5Why分析法：**
```
问题：战斗中使用技能偶发崩溃
Why1: 为什么崩溃？→ 空指针异常
Why2: 为什么空指针？→ 目标对象已被销毁但引用未清除
Why3: 为什么引用未清除？→ 异步技能释放时未检查目标有效性
Why4: 为什么未检查？→ 技能系统缺少目标有效性校验机制
Why5: 为什么缺少？→ 技能系统设计时未考虑异步场景
根因：技能系统架构设计缺陷
```

**鱼骨图(石川图)分析维度：**
- 人(People)：开发经验不足、需求理解偏差
- 机(Machine)：设备兼容性、环境差异
- 料(Material)：需求不清晰、设计文档缺失
- 法(Method)：开发流程不规范、Code Review缺失
- 环(Environment)：网络环境、第三方服务异常

## 5. 执行流程

### 5.1 准备阶段
- 确认缺陷管理工具配置(工作流、字段、权限)
- 明确当前版本的缺陷分类标准与优先级定义
- 建立缺陷模板(标题规范、必填字段、附件要求)
- 确认各角色职责(提交人、指派人、验证人)
- 同步版本发布标准(DI值、遗留Bug标准)

### 5.2 执行阶段
- 按规范提交缺陷报告
- 每日跟踪Bug状态流转，推动超时Bug
- 定期(每日/每周)进行Bug Review会议
- 对S1/S2 Bug进行即时升级处理
- 维护Bug与需求/用例的关联关系

### 5.3 分析阶段
- 按模块/类型/严重程度进行多维度统计
- 绘制缺陷收敛曲线，评估版本质量趋势
- 计算DI值，与发布标准对比
- 对高频模块进行根因分析
- 识别系统性问题(如某开发者Bug率异常高)

### 5.4 报告阶段
- 输出版本缺陷分析报告
- 汇总根因分析结果与改进建议
- 更新缺陷知识库(典型Bug案例)
- 向管理层汇报版本质量状态
- 归档历史数据用于后续版本参考

## 6. 数据指标

| 指标 | 定义 | 基线值 | 告警阈值 |
|------|------|--------|---------|
| Bug修复率 | 已修复Bug数/总Bug数 | ≥95% | <90% |
| Bug重开率 | Reopen次数/总关闭次数 | <5% | >10% |
| 平均修复时长 | 从指派到修复的平均时长 | S1<4h, S2<24h | S1>8h, S2>48h |
| DI值 | 加权缺陷密度指数 | <3 | ≥5 |
| 遗留S1/S2数 | 发布时遗留的严重Bug数 | 0 | >0 |
| Bug有效率 | 有效Bug数/总提交Bug数 | ≥90% | <80% |
| 每日新增Bug数 | 当日新提交Bug数量 | 趋势下降 | 连续3天上升 |
| 缺陷逃逸率 | 线上发现Bug数/总Bug数 | <5% | >10% |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：Bug描述不清导致反复沟通**
- 现象：开发反馈"无法复现"，来回沟通3-4轮
- 原因：复现步骤缺少前置条件、操作不够精确、缺少环境信息
- 解决：强制Bug模板(前置条件+精确步骤+实际/预期结果+环境信息+附件)，提交前自检

**坑2：严重程度与优先级混淆**
- 现象：所有Bug都标S1/P0，导致真正紧急的Bug被淹没
- 原因：团队对严重程度和优先级的定义理解不一致
- 解决：制定明确的分级标准文档，Bug Review会议统一校准，定期培训

**坑3：Bug修复引入新Bug(回归缺陷)**
- 现象：修复A功能导致B功能出问题
- 原因：模块耦合度高，修复时影响范围评估不足
- 解决：修复Bug时要求开发标注影响范围，QA针对性回归，自动化回归覆盖

**坑4：偶现Bug长期无法关闭**
- 现象：某些Bug复现率极低(1%-5%)，长期挂在列表中
- 原因：竞态条件/时序问题/特定环境触发，难以稳定复现
- 解决：增加日志埋点辅助定位，设置偶现Bug专项跟踪机制，约定超期处理策略

**坑5：版本发布前Bug数据不收敛**
- 现象：临近发布日Bug数仍在增长
- 原因：后期新增功能/需求变更引入新Bug，或前期测试不充分
- 解决：严格控制封版后的代码变更，提前设置质量门禁，必要时推迟发布
