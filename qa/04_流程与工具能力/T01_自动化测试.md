# 自动化测试

> 所属板块: 流程与工具能力 | 编号: T01 | 最后更新: 2026-03-29

---

## 1. 概述

自动化测试是游戏QA效率提升的核心手段，涵盖UI自动化、接口自动化、冒烟自动化、Monkey测试及框架搭建。通过将重复性验证工作交由脚本执行，释放人力聚焦于探索性测试和复杂场景验证。游戏自动化相比传统App自动化面临渲染引擎识别、异步加载等待、帧率敏感操作等特殊挑战，需要针对性的技术方案。

## 2. 知识体系

### 2.1 自动化测试金字塔

| 层级 | 占比建议 | 内容 | 执行速度 | 维护成本 |
|------|---------|------|---------|---------|
| 单元测试 | 70% | 函数/模块级逻辑验证 | 极快(ms级) | 低 |
| 接口/集成测试 | 20% | API协议校验、服务间交互 | 快(s级) | 中 |
| UI/E2E测试 | 10% | 全链路用户场景模拟 | 慢(min级) | 高 |

游戏项目实际比例通常为：单元测试30%、接口测试40%、UI测试30%，因为游戏逻辑高度依赖客户端表现层。

### 2.2 Page Object 模式

- 核心思想：将页面元素定位与测试逻辑分离，每个页面/面板封装为一个Page类
- 游戏适配：将游戏中的UI面板(背包、商城、战斗HUD)各自封装为Panel Object
- 三层架构：
  - BasePage层：封装通用操作(点击、滑动、等待、截图)
  - PageObject层：各面板元素定位与操作方法
  - TestCase层：业务场景组合调用

```python
# 示例：背包面板 Page Object
class BagPanel(BasePage):
    def __init__(self, driver):
        super().__init__(driver)
        self.item_list = self.find_element("bag_item_list")
        self.sort_btn = self.find_element("btn_sort")

    def use_item(self, item_name):
        item = self.find_child(self.item_list, item_name)
        self.click(item)
        self.click(self.find_element("btn_use"))
        self.wait_for_element("toast_use_success", timeout=5)
```

### 2.3 数据驱动与关键字驱动

| 驱动方式 | 原理 | 适用场景 | 优缺点 |
|---------|------|---------|--------|
| 数据驱动 | 测试逻辑不变，通过外部数据(CSV/Excel/JSON)参数化输入与预期 | 同一流程多组数据验证(如登录、充值金额) | 扩展性强，但逻辑变更需改代码 |
| 关键字驱动 | 将操作抽象为关键字(Click/Input/Assert)，通过配置文件编排 | 非技术人员编写用例、快速搭建 | 上手快，但复杂逻辑表达受限 |
| 混合驱动 | 关键字+数据驱动结合 | 大型项目 | 灵活度最高，架构复杂度也最高 |

### 2.4 游戏自动化特殊挑战

**渲染识别问题：**
- Unity/Unreal等引擎渲染的画面是GPU绘制的像素，传统DOM/View树无法识别
- 解决方案：
  - SDK注入方式：Poco/GameDriver通过SDK获取游戏对象树
  - 图像识别方式：Airtest基于OpenCV模板匹配
  - 混合方式：优先SDK定位，降级图像识别兜底

**异步等待问题：**
- 游戏中大量异步操作：资源加载、网络请求、动画播放、场景切换
- 等待策略：
  - 显式等待：轮询检测目标元素出现(推荐)
  - 隐式等待：全局超时设置(不推荐，粒度粗)
  - 自定义条件等待：监听游戏内事件(如loading完成回调)

**帧率与时序问题：**
- 低帧率设备上操作间隔需动态调整
- 战斗等实时场景需要精确的时序控制
- 建议：关键操作前插入帧稳定检测，避免在掉帧时执行输入

### 2.6 AI + MCP 自动化测试方案（前沿方向）

> 参考：Microsoft Edge QA 团队开源项目 AutoGenesis（[github.com/microsoft/AutoGenesis](https://github.com/microsoft/AutoGenesis)）

#### 核心理念：AI 只生成不执行

传统自动化的两大痛点——门槛高、维护难——可以通过 AI 辅助来突破。但关键设计决策是：**让 AI 负责理解意图和生成代码，让确定性程序负责稳定执行**。

AI 直接执行测试的风险：
- 长任务容易"走神"，忘记上下文、重复步骤、陷入循环
- 基于截图判断结果不可靠（模糊判断、无法量化、不可回溯）
- Demo 阶段可接受，生产环境致命

#### 四层架构

| 层级 | 职责 | 关键技术 | 是否依赖AI |
|------|------|---------|-----------|
| 用户层 | 用 Gherkin（BDD自然语言）描述测试场景 | Given/When/Then 语法 | 否 |
| LLM层 | 将自然语言翻译为 MCP 工具调用序列，生成 Python 步骤定义 | LLM + MCP 协议 | **是（仅此层）** |
| MCP Server层 | 跨平台能力统一封装 | PyWinauto(Windows) / Appium(macOS/iOS/Android) | 否 |
| 执行层 | Behave BDD 框架确定性执行 | Python Behave + 同步桥接 | 否 |

#### 代码生成安全机制：Preview-Confirm 三阶段

```
Step 1 — before_gen_code(feature_file, step_file)  → 预览将要修改的文件
Step 2 — gen_code(code_content)                     → AI 生成代码，人工审查
Step 3 — confirm_gen_code()                         → 确认后写入文件
```

人类始终拥有代码写入的最终控制权，防止 AI "乱写"代码。

#### 游戏QA适用场景

- 客户端 UI 回归测试（跨 Windows/macOS/iOS/Android）
- 非技术背景 QA 人员参与自动化建设
- 高频发版项目的快速回归体系搭建
- 与现有 Appium/PyWinauto 技术栈无缝集成

## 5. 执行流程

### 5.1 准备阶段
- 确认测试环境(设备、服务器、包版本)
- 更新自动化脚本至最新版本
- 检查设备连接状态与SDK版本兼容性
- 准备测试数据(账号、道具、配置)
- 确认被测版本的变更范围，调整用例集

### 5.2 执行阶段
- 触发CI流水线或本地执行测试套件
- 实时监控执行状态(通过Allure/Jenkins面板)
- 失败用例自动重试(建议最多重试2次)
- 收集执行日志、截图、视频录制
- 异常中断时自动保存现场信息

### 5.3 分析阶段
- 区分真实Bug与脚本问题(环境问题/元素变更/时序问题)
- 失败用例逐条分析，标记失败原因分类
- 对比历史执行结果，识别新增失败
- 脚本问题及时修复并回归验证
- 真实Bug提交缺陷管理系统

### 5.4 报告阶段
- 生成自动化测试报告(通过率、失败分布、耗时统计)
- 输出覆盖率数据(用例覆盖/代码覆盖/功能覆盖)
- 趋势分析(通过率变化、新增失败趋势)
- 报告同步至项目群/邮件/看板
- 维护用例健康度指标

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：图像识别在不同分辨率/机型上匹配失败**
- 现象：同一截图模板在A设备通过，B设备找不到
- 原因：分辨率差异导致缩放比例不同，模板匹配阈值过高
- 解决：使用多分辨率模板集 + 降低匹配阈值(0.7-0.8) + 优先使用Poco SDK定位

**坑2：异步加载导致元素操作失败**
- 现象：脚本执行到某步骤报"元素未找到"，但手动操作正常
- 原因：场景切换/资源加载未完成就执行了下一步操作
- 解决：所有关键操作前加显式等待，封装wait_for_element统一管理超时

**坑3：Monkey测试产生的崩溃无法复现**
- 现象：Monkey报告崩溃，但按日志操作序列无法复现
- 原因：Monkey操作速度极快，触发了正常使用不会出现的竞态条件
- 解决：降低Monkey事件间隔(--throttle 500)，开启详细日志，结合崩溃堆栈定位代码

**坑4：CI环境与本地环境执行结果不一致**
- 现象：本地全部通过，CI上部分用例失败
- 原因：CI设备性能差/网络延迟高/分辨率不同/SDK版本差异
- 解决：CI环境标准化，统一设备池配置，增加环境检查前置步骤

**坑5：自动化用例维护成本失控**
- 现象：UI频繁改版导致大量用例失败，修复速度跟不上
- 原因：用例与UI强耦合，未做好分层封装
- 解决：严格执行Page Object模式，元素定位集中管理，变更时只改Page层
