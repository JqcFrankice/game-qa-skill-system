# 服务端测试

> 所属板块: 环境与基础设施 | 编号: E01 | 最后更新: 2026-03-29

---

## 1. 概述

服务端测试是游戏QA中保障后端服务质量的核心环节，涵盖接口测试、协议验证、数据库校验、日志分析、GM工具验证、后台管理系统测试及定时任务验证。其目标是确保服务端逻辑正确、数据一致、性能达标、异常可控，为客户端提供稳定可靠的数据与服务支撑。

## 2. 知识体系

### 2.1 RESTful API 与 gRPC

- **RESTful API**: 基于HTTP协议，使用标准方法（GET/POST/PUT/DELETE）操作资源，返回JSON/XML格式数据。游戏中常用于后台管理、运营活动配置、公告推送等场景。
  - 状态码规范：200成功、400参数错误、401未授权、403禁止、404资源不存在、500服务内部错误。
  - 版本管理：URL路径版本（/api/v1/）或Header版本（Accept-Version）。
  - 幂等性要求：GET/PUT/DELETE天然幂等，POST需通过业务Token或唯一请求ID保证幂等。

- **gRPC**: 基于HTTP/2的高性能RPC框架，使用Protobuf序列化，支持四种通信模式：
  - Unary（一元调用）：请求-响应模式，适用于登录、查询等。
  - Server Streaming：服务端流式推送，适用于排行榜实时更新。
  - Client Streaming：客户端流式上传，适用于批量数据上报。
  - Bidirectional Streaming：双向流，适用于实时战斗同步。

### 2.2 Protobuf 协议

- **定义与优势**: Protocol Buffers是Google开发的二进制序列化协议，相比JSON体积减少60-80%，解析速度提升5-10倍。
- **字段规则**: required（必填，proto2）、optional（可选）、repeated（数组）、map（字典）。
- **兼容性原则**:
  - 新增字段使用新编号，不复用已删除字段编号。
  - 不修改已有字段的类型和编号。
  - 使用reserved关键字保留废弃字段编号。
- **常见问题**: 字段默认值（int32默认0、string默认空串、bool默认false）可能导致业务逻辑误判。

### 2.3 接口幂等性

- **定义**: 同一请求执行一次与执行多次产生的效果相同，不会因重复请求导致数据异常。
- **游戏场景**: 购买道具、领取奖励、充值回调等涉及资产变更的操作必须保证幂等。
- **实现方案**:
  - Token机制：客户端请求前获取唯一Token，服务端校验Token是否已消费。
  - 唯一请求ID：客户端生成requestId，服务端基于requestId去重。
  - 状态机：通过订单状态流转（待支付→已支付→已发货）防止重复处理。
  - 数据库唯一约束：利用UNIQUE INDEX防止重复插入。

### 2.4 数据库事务

- **ACID特性**: 原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。
- **隔离级别**:
  - Read Uncommitted：可读未提交数据，存在脏读。
  - Read Committed：只读已提交数据，存在不可重复读。
  - Repeatable Read（MySQL默认）：同一事务内多次读取结果一致，存在幻读。
  - Serializable：完全串行化，性能最差但最安全。
- **游戏常见事务场景**: 交易系统（扣款+发货原子操作）、背包操作（扣除材料+生成道具）、排行榜更新。
- **分布式事务**: 跨服交易需使用TCC（Try-Confirm-Cancel）、Saga模式或消息队列最终一致性方案。

### 2.5 日志级别与分析方法

- **日志级别**: TRACE < DEBUG < INFO < WARN < ERROR < FATAL。
  - DEBUG：开发调试信息，生产环境关闭。
  - INFO：关键业务流程记录（登录、支付、战斗结算）。
  - WARN：非致命异常（超时重试、降级处理）。
  - ERROR：业务错误（支付失败、数据校验不通过）。
  - FATAL：系统级故障（OOM、数据库连接池耗尽）。
- **日志分析方法**:
  - 关键字搜索：通过ERROR/Exception关键字快速定位异常。
  - 链路追踪：通过TraceID串联一次请求的完整调用链。
  - 时序分析：对比异常时间点与部署/配置变更时间。
  - 聚合统计：统计错误频率、接口耗时分布、慢查询TOP N。

## 5. 执行流程

### 第1步：准备阶段

- 确认测试环境服务端版本与待测版本一致。
- 获取最新接口文档（Swagger/Protobuf定义文件）。
- 准备测试账号（不同权限等级、不同数据状态）。
- 配置抓包工具、数据库连接、日志查看权限。
- 准备测试数据（边界值、异常值、特殊字符集）。

### 第2步：执行阶段

- 按优先级P0→P1→P2顺序执行用例。
- 接口测试：逐一验证正常流程、异常参数、边界值、并发场景。
- 数据校验：每个写操作后查询数据库确认数据一致性。
- 日志验证：关键操作后检查日志完整性和准确性。
- GM工具/后台：验证功能正确性和权限控制。
- 定时任务：通过修改系统时间或手动触发验证执行逻辑。

### 第3步：分析阶段

- 汇总接口测试通过率，标记失败用例。
- 分析数据不一致问题的根因（缓存延迟/事务问题/并发冲突）。
- 统计接口响应时间，识别性能瓶颈。
- 整理日志中发现的异常模式。

### 第4步：报告阶段

- 输出服务端测试报告，包含：用例执行统计、缺陷列表、风险评估。
- 标注阻塞性问题（P0缺陷）和建议优化项。
- 提供数据库健康度评估（慢查询数量、索引使用率）。
- 归档测试数据和日志快照，便于问题复现。

## 6. 数据指标

| 指标 | 定义 | 基线值 | 告警阈值 |
|------|------|--------|----------|
| 接口响应时间(P99) | 99%请求的响应时间 | ≤200ms | >500ms |
| 接口成功率 | 非5xx响应占比 | ≥99.9% | <99.5% |
| 数据库慢查询数 | 执行时间>1s的SQL数量/小时 | ≤5条 | >20条 |
| 缓存命中率 | Redis缓存命中次数/总查询次数 | ≥95% | <85% |
| 定时任务成功率 | 成功执行次数/总触发次数 | 100% | <95% |
| 日志ERROR数量 | 每小时ERROR级别日志条数 | ≤10条 | >50条 |
| 数据库连接池使用率 | 活跃连接数/最大连接数 | ≤60% | >85% |
| 接口幂等失败率 | 重复请求导致数据异常的比例 | 0% | >0% |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

1. **Protobuf字段默认值陷阱**: int32类型默认值为0，当业务中0是有效值时（如0级装备），无法区分"未传值"和"传了0"。解决方案：使用wrapper类型（google.protobuf.Int32Value）或增加has_xxx标记字段。

2. **缓存与数据库双写不一致**: 先更新数据库再删除缓存的方案在高并发下仍可能出现不一致。解决方案：使用延迟双删策略（更新DB→删缓存→延迟500ms再删缓存），或通过binlog订阅异步更新缓存。

3. **接口超时导致重复提交**: 客户端因网络超时未收到响应而重试，服务端实际已处理成功。解决方案：所有涉及资产变更的接口必须实现幂等，使用唯一请求ID+Redis分布式锁。

4. **定时任务在多实例部署下重复执行**: 多台服务器同时触发同一定时任务。解决方案：使用分布式锁（Redis/ZooKeeper）或分布式任务调度框架（XXL-JOB/Elastic-Job）。

5. **GM工具操作无审计日志**: GM发放道具后出现纠纷无法追溯。解决方案：所有GM操作必须记录操作人、操作时间、操作内容、影响玩家，且日志不可篡改。
