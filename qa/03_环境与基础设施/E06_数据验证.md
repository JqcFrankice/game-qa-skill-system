# 数据验证

> 所属板块: 环境与基础设施 | 编号: E06 | 最后更新: 2026-03-29

---

## 1. 概述

数据验证是游戏运营决策的质量基石，涵盖埋点校验、数据上报、BI看板、漏斗分析及AB测试等核心领域。准确的数据采集和分析能力直接影响运营策略的有效性，错误的数据可能导致错误的决策，造成不可逆的损失。QA在数据链路中承担着从采集端到展示端的全链路质量保障职责。

## 2. 知识体系

### 2.1 埋点设计规范

- **埋点分类**:
  - 客户端埋点：记录用户在客户端的行为（点击、页面浏览、操作路径）。
  - 服务端埋点：记录服务端业务事件（登录、支付、道具变更）。服务端埋点更可靠，不受客户端环境影响。
  - 全埋点/无埋点：SDK自动采集所有可交互元素的点击和页面浏览事件，无需手动埋点。
- **埋点三要素**: Who（用户标识）+ When（时间戳）+ What（事件名+属性）。
- **命名规范**:
  - 事件名：模块_动作（如`shop_buy`、`battle_start`、`login_success`）。
  - 属性名：snake_case格式，语义明确（如`item_id`、`item_count`、`currency_type`）。
  - 避免：拼音命名、含义模糊的缩写、频繁变更事件名。
- **埋点文档**: 每个埋点需记录事件名、触发时机、属性列表（名称/类型/是否必填/示例值/说明）、负责人、上线版本。
- **埋点生命周期**: 需求评审 → 埋点设计 → 开发实现 → QA验证 → 上线监控 → 下线清理。

### 2.2 数据采集SDK原理

- **SDK架构**: 事件采集层 → 本地缓存层 → 上报策略层 → 网络传输层。
- **本地缓存**: 事件先写入本地数据库（SQLite/文件），防止网络不可用时数据丢失。
- **上报策略**:
  - 实时上报：关键事件（支付、崩溃）立即上报。
  - 批量上报：普通事件累积到一定数量（如50条）或时间间隔（如60秒）后批量上报。
  - 启动上报：App启动时上报上次未成功发送的缓存数据。
- **数据压缩**: 上报前对数据进行gzip压缩，减少网络传输量。
- **用户标识**:
  - 设备ID：IDFA(iOS)/GAID(Android)/OAID(国内Android)。
  - 用户ID：登录后的账号ID。
  - 匿名ID：未登录时的临时标识，登录后与用户ID关联（ID Mapping）。
- **主流SDK**: 神策分析、GrowingIO、Firebase Analytics、Adjust、AppsFlyer。

### 2.3 数据仓库ETL

- **ETL流程**: Extract（抽取）→ Transform（转换）→ Load（加载）。
  - Extract：从多个数据源（游戏日志、支付系统、第三方平台）抽取原始数据。
  - Transform：数据清洗（去重、去空、格式统一）、数据转换（时区转换、枚举映射）、数据聚合（按天/按用户汇总）。
  - Load：将处理后的数据加载到数据仓库（Hive/ClickHouse/BigQuery）。
- **数据分层**:
  - ODS（Operational Data Store）：原始数据层，保留原始格式。
  - DWD（Data Warehouse Detail）：明细数据层，清洗后的标准化数据。
  - DWS（Data Warehouse Summary）：汇总数据层，按主题聚合。
  - ADS（Application Data Store）：应用数据层，直接供BI看板使用。
- **数据质量校验**: 完整性（数据量对比）、准确性（抽样校验）、及时性（延迟监控）、一致性（跨表对账）。

### 2.4 AB测试统计学基础

- **假设检验**:
  - 零假设（H0）：实验组与对照组无显著差异。
  - 备择假设（H1）：实验组与对照组存在显著差异。
  - 显著性水平（α）：通常取0.05，即5%的概率犯第一类错误（假阳性）。
  - 统计功效（1-β）：通常要求≥80%，即80%的概率检测到真实差异。
- **样本量计算**: 基于预期效果量（MDE, Minimum Detectable Effect）、基线转化率、显著性水平和统计功效计算所需样本量。样本量不足会导致实验结论不可靠。
- **常用检验方法**:
  - Z检验/t检验：比较两组均值差异（如ARPU）。
  - 卡方检验：比较两组比例差异（如转化率、留存率）。
  - Mann-Whitney U检验：非正态分布数据的非参数检验。
- **实验陷阱**:
  - 窥探效应（Peeking）：实验未达到预定样本量就提前查看结果并做决策。
  - 辛普森悖论：整体趋势与分组趋势相反，需按维度拆分分析。
  - 新奇效应：用户对新功能的短期兴趣导致指标虚高，需观察足够长时间。
  - 多重比较问题：同时测试多个指标时需进行Bonferroni校正。

## 5. 执行流程

### 第1步：准备阶段

- 获取最新埋点文档和数据字典。
- 配置抓包工具，确认能拦截数据上报请求。
- 获取BI看板和数据仓库的查询权限。
- 准备AB测试实验配置和样本量计算。

### 第2步：执行阶段

- 逐一验证每个埋点的触发时机、事件名和属性完整性。
- 验证数据上报的格式、加密、重试和性能。
- 对比BI看板数据与原始数据的一致性。
- 验证漏斗分析的步骤定义和转化率计算。
- 验证AB测试的分组随机性、稳定性和指标统计。

### 第3步：分析阶段

- 汇总埋点校验结果，标记缺失/错误的埋点。
- 分析数据链路的延迟和丢失率。
- 评估BI看板数据的准确性和时效性。
- 验证AB测试结论的统计显著性。

### 第4步：报告阶段

- 输出数据验证报告，包含埋点覆盖率、数据准确率。
- 列出数据质量问题和修复建议。
- 提供AB测试结果的统计学评估。
- 更新埋点文档和数据字典。

## 6. 数据指标

| 指标 | 定义 | 基线值 | 告警阈值 |
|------|------|--------|----------|
| 埋点覆盖率 | 已验证通过的埋点数/需求埋点总数 | 100% | <95% |
| 数据上报成功率 | 成功上报事件数/触发事件总数 | ≥99.5% | <98% |
| 数据延迟(实时) | 事件触发到数据可查询的时间 | ≤5分钟 | >15分钟 |
| 数据延迟(T+1) | 离线数据就绪时间 | 次日10:00前 | 次日12:00后 |
| BI看板数据准确率 | 看板数据与原始数据一致的指标占比 | 100% | <98% |
| AB实验分组偏差 | 各组用户数最大偏差/期望用户数 | ≤5% | >10% |
| 埋点属性错误率 | 属性值异常的事件数/总事件数 | ≤0.1% | >1% |
| ETL任务成功率 | ETL任务成功执行次数/总调度次数 | ≥99% | <95% |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

1. **埋点漏埋导致数据缺失**: 新功能上线后发现关键埋点未实现，无法分析用户行为。解决方案：将埋点验证纳入功能测试的必检项，上线前通过抓包逐一确认所有埋点。

2. **时区不一致导致数据统计错误**: 客户端上报UTC时间，服务端按本地时间统计，导致跨天数据错乱。解决方案：全链路统一使用UTC时间戳，展示层按用户时区转换。

3. **AB测试样本量不足就下结论**: 实验运行3天就因为"看起来有差异"而做出决策，实际差异不显著。解决方案：实验前计算所需样本量，严格按照预定计划运行，使用序贯检验方法如果需要提前决策。

4. **埋点属性值枚举不统一**: 同一属性在不同版本使用不同的枚举值（如currency_type: "gold"/"Gold"/"GOLD"/1），导致统计混乱。解决方案：建立统一的数据字典，SDK层面做枚举值校验和标准化。

5. **数据上报影响游戏性能**: 大量埋点同时触发导致主线程卡顿。解决方案：数据采集和上报在独立线程/协程中执行，设置上报频率上限，非关键事件降采样。
