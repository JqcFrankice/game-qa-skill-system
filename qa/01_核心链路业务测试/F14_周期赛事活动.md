# 周期赛事活动测试规范

> 所属板块: 核心链路业务测试 | 编号: F14 | 最后更新: 2026-03-31
> 本Skill为周期性、竞技性或按赛季组织的活动的专项通用测试规范，可在生成测试用例时作为补充规则动态加载。

---

## 1. 概述

### 1.1 Skill定位

本Skill专门针对包含多阶段状态流转、按周期/赛季执行，且强依赖服务器分组/跨服机制的**竞技/赛事/周期性活动**系统。如：阵营竞技、排位赛、巅峰赛、跨服战、门派战等。

### 1.2 核心测试目标

- 确保整个活动周期从开启、进行、结算到重置的每个阶段的**状态正确切换**。
- 保证**赛程的极端边界条件**（如人数不足、轮空、异步等待）不会导致逻辑卡死。
- 验证分组/跨服条件，以及相关的奖励发放节点的精准度。

## 2. 知识体系

### 2.1 专项补充规则 (强制发散要求)

在生成**竞技/赛事/周期性活动**的测试用例时，为避免用例数量锐减、覆盖不全、缺乏多用户视角（如 AI 生成只关注单机主流程），必须**强制裂变**以下 8 个维度的测试点，且每个维度必须拆分为独立的 TC 编号，严禁合并：

1. **周期流转与状态切片 (State Machine)**：不仅要覆盖“活动开启、预赛、淘汰、竞猜、结算”，**必须**覆盖每个阶段的“未到时间”、“准点瞬间”、“阶段进行中”和“阶段结束瞬间” 4 个时刻的表现与拦截。
2. **多角色与旁观者视角 (Multi-Role Perspective - 核心补充)**：赛事绝不仅是参赛者的逻辑！必须强制生成以下并发场景：
   - **双边实时同步**：玩家 A 在准备阶段更换阵容/外显，玩家 B 的匹配界面必须实时刷新战力与头像。
   - **异步操作隔离**：玩家 A 点击“跳过战斗”进入结算，玩家 B 仍在观看战斗，A 的行为不能影响 B 的进程。
   - **旁观者界面**：未参赛玩家（或已淘汰玩家）打开活动界面的表现；赛程图高亮/置灰状态的实时更新；竞猜中/战斗中/已结束不同状态下，观战按钮的点击反馈。
   - **战报高并发并发拉取**：必须模拟多名玩家同时查看同一场热门赛事的战报，验证战斗服与回放机制的承载力。

3. **参赛人数极端边界 (Edge Cases)**：必须穷举：① 全服 0 人报名；② 报名人数不足开启预赛（直接结束）；③ 单一分组/阵营人数极少（跳过某轮次直接晋级）；④ 奇数人数导致的单轮轮空、连续轮空积分处理。

4. **多阵营/多组别异步等待时序 (Asynchronous Wait)**：A 组别/阵营因人数少提前打完（如只需打 8 场），B 组别需打满 20 场。必须验证 A 组玩家的 UI 是否处于正确的“等待同步”倒计时状态，且系统不能卡死。

5. **阶段性数据存库与异常注入 (Phased Persistence - 极高颗粒度要求)**：**严禁只写一条笼统的“断线重连”或“宕机”用例**。对于周期赛事，必须按“上下线（内存与缓存）”与“重启服务器（DB落地）”在以下关键节点进行分离验证：
   - **存库白名单**：报名状态、分配阵营、已交手记录列表（防重复匹配）、淘汰赛对阵表、当轮积分、晋级/淘汰状态、押注对象与资格、各档位奖励领取状态。
   - **服务器细分宕机 (Crash Matrix)**：必须拆分 **Game服宕机**（影响逻辑流转）、**Battle服宕机**（战斗中途崩溃的超时兜底与结果下发）、**Cross服宕机**（跨服匹配期的中断与恢复）。
   - **弱网与断线**：必须覆盖报名瞬间、切界面瞬间、战斗倒计时最后一秒、点击下注的瞬间。

6. **底层协议拦截与防刷验证 (Protocol Anti-Cheat)**：每个写操作接口（报名、修改阵容、匹配、战斗结算、下注竞猜）都**必须**生成独立的协议拦截用例：
   - 在非当前阶段（如报名期外）调用对应接口。
   - 调用接口修改已落定的状态（如已报名尝试取消，已下注尝试更改目标）。
   - 越权接口：调用接口强行设定匹配已对战过的玩家、强行给轮空玩家安排对手、让已淘汰玩家调用晋级发奖接口。

7. **积分、排名与排序兜底规则**：除了验证正常胜负加减分外，**必须**验证：① 连败导致的 0 分下限保护（不出现负分）；② 积分完全相同时的排序兜底策略（如同分看战力，同战力看 UID，严禁出现并列导致晋级名额超发）。

8. **精确的奖励发放、拍卖联动与跨服分组**：
   - **跨服与合服判定**：首次玩法必须验证本服隔离；第二次开启必须验证跨服生效；周一合服/重新分组后，必须验证赛事分组锁定机制不被破坏；UI必须附加服务器前缀（如 S1.张三）。
   - **全服拍卖基准**：验证全服0人参与时的“保底商品产出”；跨服环境下的商品产出数量是否仅根据“本服参与人数”隔离计算；参与者与未参与者的分红差异。
   - **发奖时机与幂等**：被淘汰时的“即时发奖”、全活动结束时的“统一发奖”，邮件与界面领取必须做二次防刷校验。

## 3. 测试用例框架

### 3.1 核心用例设计

| ID | 场景分类 | 测试步骤 | 预期结果 | 优先级 |
|----|----------|----------|----------|--------|
| TC-F14-001 | 周期流转 | 等待系统到达报名期、预赛期、淘汰赛期切换时间节点 | 每个阶段状态无缝衔接，界面与操作权限同步变更 | P0 |
| TC-F14-002 | 人数极端边界 | 单服参与人数无法凑齐一场基础比赛，或者淘汰赛某组别为奇数 | 触发人数不足轮空晋级规则，或系统给出补偿/提前结束逻辑并正常运转 | P0 |
| TC-F14-003 | 异步等待 | A阵营已决出胜者，B阵营仍在比赛中 | A阵营玩家进入“等待同步”界面并显示倒计时，直到B结束 | P0 |
| TC-F14-004 | 断线恢复 | 在比赛匹配中或倒计时读秒阶段断网/强杀进程后重连 | 界面能恢复到当前的精确倒计时和正确等待/比赛状态 | P1 |
| TC-F14-005 | 分组与门槛 | 新服（未满特定天数）玩家尝试进入跨服匹配入口 | 提示“开服天数不足”或系统隐藏该跨服入口 | P0 |
| TC-F14-006 | 奖励节点 | 一名玩家在8进4被淘汰，确认奖励发放情况 | 在“被淘汰瞬间”或“整个淘汰赛结束统一时刻”通过邮件/发放到账对应的参与奖与名次奖 | P0 |

## 4. 踩坑经验库

| 问题描述 | 根因分析 | 规避方案 |
|----------|----------|----------|
| 赛事极少人数逻辑卡死 | 仅按理想人数流程生成逻辑，漏测真实环境中报名不足的边界 | 穷举极端人数/异步等待/轮空情况。强制设计轮空直接晋级与跳过赛程阶段的用例。 |
| 跨服分组门槛遗漏 | 把注意力放在核心玩法上，忽略了运营前置开服时间条件 | 跨服玩法必须包含“按服务器分组”、“开服时间限制”、“合服影响确认”三个前置用例模块。 |
| 多阵营进度不同步导致UI卡死 | 各组进度不一致导致的异步等待状态设计缺陷 | 必须编写“某一分组提前结束”时的异步等待与UI同步更新用例。 |
| 宕机与重启数据丢失/重发 | 仅停留在业务主流程，未将宕机精细化为Game服/Battle服/Cross服，缺少DB存库校验。 | 必须把宕机验证拆细。例如：淘汰赛发奖前一秒Game服宕机，恢复后必须验证淘汰奖正常发放；匹配完成但未出结果时Battle宕机，需验证兜底结果。 |
| 旁观者高并发拉取战报崩溃 | 忽略了周期赛事的明星玩家对局，会同时有全服数千人同时拉取同一份战报文件。 | 用例框架必须补充一条：“配置多个玩家同时拉取同一场决胜战报并验证高并发表现（防惊群效应）”。 |
| 全服拍卖产出与分红计算异常 | 漏测跨服情况下的本服/跨服参与人数分离计算，漏测 0 参与时的保底掉落机制。 | 增加专门针对“拍卖与赛事联动”的测试点：① 完全无人参与是否触发保底。② 跨服参与人数仅计算本服活跃。 |
