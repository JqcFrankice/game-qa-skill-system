# 项目特性

> 所属板块: 前置基础 | 编号: G00 | 最后更新: 2026-03-30

---

## 1. 概述

项目特性是整个QA技能体系的前置基础，定义了"我们在测什么"。所有后续Skill（F01-F12业务测试、S01-S09专项测试、T01-T07工具流程等）的执行策略、用例设计、自动化方案都应以项目特性为输入。一个清晰的项目特性档案，是测试策略制定、用例生成、自动化脚本编写的起点。

**核心准则：如果要生成用例，必须首先在 `openspec` 目录下寻找指定的功能文件（优先级：策划案/Excel/md > 设计/design > 方案/proposal）。**

**核心价值：**
- 为用例生成提供上下文（项目类型决定测试重点）
- 为自动化测试提供基础设施（GM指令是自动化的前置条件）
- 为团队协作提供共识（新人快速理解项目全貌）
- 为AI辅助测试提供Prompt上下文（项目特性 → Gherkin场景生成）

### 4.2 风格对测试的影响

| 风格特征 | 测试影响 | 关注点 |
|---------|---------|--------|
| 写实画风 | 性能压力大，资源量大 | GPU/内存/加载时间/LOD切换 |
| 二次元 | 角色表现力要求高 | 动画流畅度/表情/Live2D |
| 快节奏战斗 | 帧率敏感、操作时序严格 | 输入延迟/帧率稳定/打击感 |
| 开放世界 | 大世界加载、物理交互多 | 流式加载/物理碰撞/边界测试 |
| 多人社交 | 并发压力、聊天安全 | 压力测试/敏感词/消息同步 |
| 虚拟摇杆 | 操作精度、适配问题 | 不同屏幕尺寸/误触/灵敏度 |

## 5. GM指令集合

### 5.1 概述

GM指令（Game Master Commands）是测试环境中用于快速构造测试条件的命令集合。它是自动化测试的基础设施——没有GM指令，大量测试场景无法高效构造。

**GM指令对自动化的价值：**
- 快速构造前置条件（跳过漫长的手动操作）
- 精确控制游戏状态（设置等级、金币、时间等）
- 可重复的测试环境（每次执行前重置到相同状态）
- 与BDD场景的Given步骤天然对应

### 5.2 GM指令分类标准

| 分类 | 用途 | 示例指令 | 对应Skill |
|------|------|---------|----------|
| 账号类 | 账号状态控制 | `gm set_level 60` / `gm set_vip 10` | F01/F10 |
| 货币类 | 经济数据控制 | `gm add_gold 99999` / `gm add_diamond 1000` | F04/F09 |
| 道具类 | 物品发放/清除 | `gm add_item 10001 99` / `gm clear_bag` | F05 |
| 任务类 | 任务进度控制 | `gm complete_quest 1001` / `gm reset_quest` | F06 |
| 战斗类 | 战斗状态控制 | `gm god_mode on` / `gm set_hp 99999` | F03 |
| 场景类 | 场景跳转 | `gm teleport 100 200` / `gm enter_scene 3001` | E01 |
| 时间类 | 时间控制 | `gm set_time 2026-01-01` / `gm skip_cd` | F11 |
| 系统类 | 系统开关 | `gm open_activity 5001` / `gm set_server_time` | F11/E02 |
| 调试类 | 调试辅助 | `gm show_fps` / `gm show_hitbox` / `gm log_level debug` | S01/T05 |

### 5.3 GM指令登记模板

```yaml
# === GM指令登记 ===
# 每条指令需登记以下信息，用于自动化脚本引用

- command: "gm add_gold {amount}"
  category: "货币类"
  description: "增加金币"
  parameters:
    - name: amount
      type: int
      range: "1-999999999"
  precondition: "角色已创建"
  side_effect: "触发金币变更事件，可能触发成就"
  automation_usage: "Given步骤：构造充足金币的前置条件"
  available_env: ["dev", "qa", "staging"]  # 可用环境
```

### 5.4 GM指令与自动化测试的映射

```gherkin
# GM指令在BDD场景中的典型用法

Feature: 商城购买流程
  Background:
    Given 通过GM指令设置玩家等级为60
    And 通过GM指令添加10000钻石

  Scenario: 购买限购商品
    When 打开商城页面
    And 购买"限定礼包"
    Then 钻石减少对应金额
    And 背包中出现礼包道具
```

对应的自动化步骤定义：
```python
@given('通过GM指令设置玩家等级为{level:d}')
def step_set_level(context, level):
    context.gm.execute(f"gm set_level {level}")
    context.gm.wait_for_effect(timeout=5)

@given('通过GM指令添加{amount:d}钻石')
def step_add_diamond(context, amount):
    context.gm.execute(f"gm add_diamond {amount}")
    context.gm.wait_for_effect(timeout=5)
```

### 5.5 GM指令管理规范

| 规范项 | 要求 | 说明 |
|--------|------|------|
| 命名规范 | `gm {动词}_{名词} {参数}` | 统一格式，便于自动化解析 |
| 权限控制 | 仅限测试环境，生产环境禁用 | 防止线上误操作 |
| 版本同步 | 每次版本更新同步GM指令变更 | 避免自动化脚本因指令变更失败 |
| 文档维护 | 新增/修改/废弃指令必须更新本文档 | 保持指令集合与实际一致 |
| 返回值 | 每条指令返回执行结果（成功/失败/原因） | 自动化脚本需要判断执行结果 |
| 幂等性 | 同一指令重复执行结果一致 | 保证自动化重试的可靠性 |

## 7. 执行流程

### 7.1 项目特性建档（项目启动时）
- 填写项目类型登记模板（2.2节）
- 填写项目基本信息模板（3.1节）
- 填写核心系统清单（3.2节）
- 填写美术与交互风格（4.1节）
- 根据品类映射表确定高优Skill清单

### 7.2 GM指令收集（研发阶段持续）
- 与开发团队对齐GM指令清单
- 按5.3模板逐条登记
- 验证每条指令在测试环境可用
- 建立GM指令与测试场景的映射关系

### 7.3 定期维护（版本迭代时）
- 每个版本检查GM指令变更（新增/修改/废弃）
- 更新项目特性档案（新系统上线、平台扩展等）
- 同步自动化脚本中的GM指令引用

## 8. 踩坑经验库

### 高频踩坑 TOP 5

**坑1：GM指令在不同环境行为不一致**
- 现象：QA环境GM指令正常，Staging环境执行无效
- 原因：Staging环境GM权限被关闭或指令版本不同步
- 解决：建立环境-指令可用性矩阵，自动化脚本执行前先验证GM可用性

**坑2：GM指令有副作用导致测试结果不准**
- 现象：用GM加金币后，触发了成就系统奖励，导致金币数量与预期不符
- 原因：GM指令触发了游戏内事件链
- 解决：登记每条GM指令的副作用（5.3模板中的side_effect字段），测试用例中考虑副作用

**坑3：项目类型判断错误导致测试策略偏差**
- 现象：把"开放世界+MMO"项目按纯MMO策略测试，遗漏了大世界加载相关问题
- 原因：项目品类有交叉，只按主品类制定策略
- 解决：项目类型支持多标签，测试策略取所有相关品类的Skill并集

**坑4：GM指令文档与实际不同步**
- 现象：自动化脚本调用GM指令失败，排查发现指令参数已变更
- 原因：开发修改了GM指令但未通知QA
- 解决：GM指令变更纳入版本发布checklist，CI中增加GM指令可用性检查

**坑5：忽略项目风格对性能测试基线的影响**
- 现象：用通用性能基线评估写实风格游戏，大量误报
- 原因：写实风格GPU/内存消耗天然高于卡通风格
- 解决：性能基线按项目风格分级设定，不使用统一阈值
