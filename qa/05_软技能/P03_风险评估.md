# 风险评估

> 所属板块: 软技能 | 编号: P03 | 最后更新: 2026-03-29

---

## 1. 概述

风险评估是QA在游戏项目中保障上线质量的关键能力。它要求测试人员能系统性地识别、分析、量化和应对各类质量风险，从"被动发现问题"转变为"主动预防问题"。优秀的风险评估能力能帮助团队在有限的时间和资源下做出最优的测试策略决策，确保关键风险得到充分覆盖。

**核心价值**：
- 在资源有限时聚焦最高风险区域，提升测试ROI
- 为上线决策提供量化的风险依据
- 通过灰度策略和应急预案降低上线事故影响
- 建立团队的风险意识和预防文化

## 2. 知识体系

### 2.1 风险识别方法

**风险来源分类**：

| 风险来源 | 典型风险 | 识别方法 |
|----------|----------|----------|
| 需求变更 | 需求频繁变更导致测试覆盖不足 | 跟踪需求变更记录，评估变更影响 |
| 技术实现 | 新技术/架构引入的未知风险 | 技术方案评审，关注技术债务 |
| 时间压力 | 测试时间被压缩导致覆盖不足 | 评估测试工作量与可用时间的差距 |
| 人员变动 | 关键人员离职/请假导致知识断层 | 关注人员稳定性，推动知识文档化 |
| 外部依赖 | 第三方SDK/服务不稳定 | 梳理外部依赖清单，关注SLA |
| 历史问题 | 老代码/技术债务引发的回归问题 | 分析历史Bug数据，识别高频问题模块 |
| 运营活动 | 活动配置错误/时间冲突 | 活动配置检查清单，交叉验证 |

**风险识别技术**：
- 头脑风暴：团队集体识别潜在风险
- 检查清单法：基于历史经验的结构化检查
- 因果分析：从已知问题反推潜在风险
- 假设分析（What-If）：假设某个环节出问题，推演影响
- 历史数据分析：分析过往版本的Bug分布和线上事故

### 2.2 风险矩阵（概率 × 影响）

**风险等级评估矩阵**：

| | 影响-极低(1) | 影响-低(2) | 影响-中(3) | 影响-高(4) | 影响-极高(5) |
|---|---|---|---|---|---|
| 概率-极高(5) | 中(5) | 高(10) | 高(15) | 极高(20) | 极高(25) |
| 概率-高(4) | 低(4) | 中(8) | 高(12) | 高(16) | 极高(20) |
| 概率-中(3) | 低(3) | 中(6) | 中(9) | 高(12) | 高(15) |
| 概率-低(2) | 低(2) | 低(4) | 中(6) | 中(8) | 高(10) |
| 概率-极低(1) | 低(1) | 低(2) | 低(3) | 低(4) | 中(5) |

**风险等级对应策略**：

| 风险等级 | 分值范围 | 应对策略 |
|----------|----------|----------|
| 极高 | 20-25 | 必须在上线前消除或降级；需要专项测试和应急预案 |
| 高 | 10-16 | 优先处理；需要充分测试覆盖和规避方案 |
| 中 | 5-9 | 正常跟踪；纳入测试计划，有基本应对措施 |
| 低 | 1-4 | 接受风险；记录并监控，不需要专项投入 |

### 2.3 变更影响分析

**变更影响评估框架**：

```
变更内容
├── 直接影响（被修改的模块/功能）
│   ├── 功能正确性
│   ├── 数据兼容性
│   └── 接口变化
├── 间接影响（依赖该模块的其他功能）
│   ├── 上游调用方
│   ├── 下游被调用方
│   └── 共享数据/状态
└── 潜在影响（可能受影响但不确定的区域）
    ├── 性能变化
    ├── 并发场景
    └── 边界条件
```

**影响面分析方法**：

| 方法 | 适用场景 | 操作方式 |
|------|----------|----------|
| 代码diff分析 | 代码变更 | 分析代码改动涉及的文件/函数/接口 |
| 依赖关系图 | 系统级变更 | 梳理模块间调用关系，识别影响链路 |
| 数据流分析 | 数据相关变更 | 追踪数据的产生/传递/消费路径 |
| 历史回归分析 | 高频变更模块 | 分析该模块历史变更引发的回归Bug |
| 配置影响分析 | 配置/数值变更 | 检查配置项的引用位置和影响范围 |

### 2.4 灰度发布策略

**灰度维度**：

| 灰度维度 | 说明 | 适用场景 |
|----------|------|----------|
| 服务器灰度 | 按服务器分批开放 | 多服架构的游戏 |
| 用户灰度 | 按用户ID/特征分批 | 需要精确控制的功能 |
| 渠道灰度 | 按渠道/平台分批 | 多渠道发行的游戏 |
| 地区灰度 | 按地区/国家分批 | 全球发行的游戏 |
| 功能开关 | 通过配置控制功能开关 | 需要快速回退的功能 |

**灰度发布流程**：
```
阶段1: 内部灰度（1-5%）
├── 内部员工/测试账号
├── 验证基本功能正确性
└── 持续时间: 1-2天

阶段2: 小规模灰度（5-20%）
├── 选择代表性用户群
├── 监控核心数据指标
├── 收集用户反馈
└── 持续时间: 2-3天

阶段3: 大规模灰度（20-50%）
├── 扩大用户范围
├── 验证大规模场景
├── 确认数据指标稳定
└── 持续时间: 1-2天

阶段4: 全量发布（100%）
├── 全量放开
├── 持续监控24-48小时
└── 保留回滚能力
```

### 2.5 回滚方案设计

**回滚方案要素**：

| 要素 | 内容 | 注意事项 |
|------|------|----------|
| 触发条件 | 什么情况下触发回滚 | 明确量化指标（如崩溃率>5%） |
| 回滚范围 | 回滚哪些内容 | 代码/配置/数据分别处理 |
| 回滚步骤 | 具体操作步骤 | 步骤要详细到可执行 |
| 数据处理 | 回滚期间产生的数据如何处理 | 最复杂的部分，需要提前设计 |
| 时间预估 | 回滚需要多长时间 | 包括操作时间和生效时间 |
| 验证方法 | 如何确认回滚成功 | 明确验证步骤和检查点 |
| 通知机制 | 回滚时通知哪些人 | 包括内部团队和外部用户 |

### 2.6 应急响应流程

**应急响应等级**：

| 等级 | 定义 | 响应时间 | 处理要求 |
|------|------|----------|----------|
| P0-紧急 | 大面积不可用/资产损失 | 5分钟内响应 | 全员待命，优先恢复服务 |
| P1-严重 | 核心功能异常/部分用户受影响 | 15分钟内响应 | 相关人员到位，1小时内出方案 |
| P2-一般 | 非核心功能异常/少量用户受影响 | 30分钟内响应 | 正常流程处理，当日内解决 |
| P3-轻微 | 体验问题/个别用户反馈 | 2小时内响应 | 记录跟踪，排入正常迭代 |

**应急响应SOP**：
```
发现问题 → 初步判断等级 → 通知相关人员 → 组建应急小组
    → 定位问题 → 制定方案（修复/回滚/降级）
    → 执行方案 → 验证恢复 → 通知解除
    → 事后复盘 → 更新预案
```

## 4. 实践框架/检查清单

### 4.1 版本风险评估检查清单

- [ ] 识别本版本所有变更点（需求变更/代码变更/配置变更）
- [ ] 对每个变更点进行影响面分析
- [ ] 使用风险矩阵对识别的风险进行评级
- [ ] 高风险项制定专项测试方案
- [ ] 极高风险项制定应急预案和回滚方案
- [ ] 风险评估结果同步给项目组所有成员
- [ ] 确认灰度发布策略和节奏

### 4.2 上线风险检查清单

- [ ] 所有P0 Bug已修复并验证
- [ ] P1 Bug已修复或有明确的规避方案
- [ ] 回归测试覆盖所有高风险变更
- [ ] 性能测试达标（帧率/延迟/内存/崩溃率）
- [ ] 灰度方案已确认（灰度范围/节奏/监控指标）
- [ ] 回滚方案已确认（触发条件/操作步骤/数据处理）
- [ ] 应急预案已确认（值班人员/响应流程/通知机制）
- [ ] 各方已签字确认上线

### 4.3 回归范围评估检查清单

- [ ] 分析代码diff，确定直接影响模块
- [ ] 梳理模块依赖关系，确定间接影响范围
- [ ] 检查数据库/配置变更的影响
- [ ] 参考历史回归Bug，补充高频回归区域
- [ ] 确定回归测试优先级（P0>P1>P2）
- [ ] 评估回归测试工作量与可用时间
- [ ] 如时间不足，与项目组沟通裁剪方案

### 4.4 应急预案检查清单

- [ ] 明确应急响应等级定义和触发条件
- [ ] 确认值班人员和联系方式
- [ ] 确认应急通讯群/频道已建立
- [ ] 回滚脚本/工具已准备并测试
- [ ] 服务降级方案已确认
- [ ] 玩家公告/补偿方案模板已准备
- [ ] 应急演练已执行（至少桌面推演）

## 5. 执行流程

### 5.1 版本风险评估SOP

```
Step 1: 信息收集
├── 收集本版本需求变更清单
├── 收集代码变更清单（diff/commit log）
├── 收集配置/数值变更清单
└── 了解技术方案和架构变化

Step 2: 风险识别
├── 按风险来源分类逐项检查
├── 使用检查清单辅助识别
├── 参考历史版本的风险和事故
└── 与开发/策划沟通确认潜在风险

Step 3: 风险评估
├── 对每个风险评估概率（1-5）
├── 对每个风险评估影响（1-5）
├── 计算风险值并确定等级
└── 输出风险矩阵图

Step 4: 风险应对
├── 极高风险: 制定消除/降级方案
├── 高风险: 制定专项测试+应急预案
├── 中风险: 纳入测试计划正常覆盖
└── 低风险: 记录并监控

Step 5: 跟踪闭环
├── 定期更新风险状态
├── 新增风险及时补充
├── 风险消除后标记关闭
└── 版本结束后复盘风险评估准确性
```

### 5.2 上线发布SOP

```
Step 1: 上线前准备（T-2天）
├── 完成风险评估报告
├── 确认灰度发布方案
├── 确认回滚方案
├── 确认应急预案和值班安排
└── 组织上线评审会

Step 2: 灰度发布（T日）
├── 按计划执行灰度发布
├── 实时监控核心指标
├── 关注玩家反馈渠道
└── 每个灰度阶段输出监控报告

Step 3: 全量发布
├── 灰度数据达标后全量放开
├── 持续监控24-48小时
├── 保持应急响应状态
└── 确认无异常后解除应急状态

Step 4: 上线后总结
├── 输出上线质量报告
├── 记录上线过程中的问题
├── 更新风险库和应急预案
└── 复盘风险评估的准确性
```

### 5.3 线上事故应急SOP

```
Step 1: 发现与报告（5min内）
├── 确认问题现象和影响范围
├── 初步判断应急等级
├── 通知应急响应群
└── 指定事故指挥官

Step 2: 定位与决策（30min内）
├── 组建应急小组（QA+开发+运维）
├── 快速定位问题原因
├── 制定处理方案（修复/回滚/降级）
└── 事故指挥官决策执行方案

Step 3: 执行与验证
├── 执行处理方案
├── 验证问题是否解决
├── 确认无新问题引入
└── 监控恢复后的数据指标

Step 4: 通知与善后
├── 内部通知事故解除
├── 必要时发布玩家公告
├── 制定补偿方案（如需要）
└── 安排事后复盘

Step 5: 复盘与改进
├── 48小时内组织事故复盘
├── 输出事故复盘报告（时间线/根因/改进项）
├── 跟踪改进项落地
└── 更新应急预案和风险库
```

## 7. 踩坑经验库

### 高频踩坑 TOP 6

**坑1: 只关注新功能风险，忽视回归风险**
- 场景：新功能测试充分，但老功能因代码改动出现回归Bug
- 原因：变更影响分析不充分，回归范围评估遗漏
- 对策：每次变更必须做影响面分析，高频回归区域纳入自动化

**坑2: 灰度比例过大，问题扩散快**
- 场景：首批灰度就放了30%用户，发现问题时已影响大量玩家
- 原因：灰度策略过于激进，缺乏阶梯式放量
- 对策：严格执行阶梯灰度（1%→5%→20%→50%→100%），每阶段充分观察

**坑3: 回滚方案未考虑数据处理**
- 场景：代码回滚了，但灰度期间玩家产生的数据无法兼容
- 原因：回滚方案只考虑了代码层面，未考虑数据层面
- 对策：回滚方案必须包含数据处理方案，上线前进行回滚演练

**坑4: 应急预案停留在纸面，未实际演练**
- 场景：线上事故发生时，按预案操作发现步骤不通/权限不足
- 原因：预案编写后未进行实际演练验证
- 对策：重要版本上线前至少进行一次桌面推演，关键操作实际验证

**坑5: 风险评估一次性完成，不持续更新**
- 场景：版本初期做了风险评估，但后续需求变更引入的新风险未纳入
- 原因：风险评估被当作一次性工作而非持续过程
- 对策：风险评估随版本迭代持续更新，每次需求/技术变更后重新评估

**坑6: 过度依赖灰度，忽视上线前测试**
- 场景："反正有灰度兜底"，测试不充分就上灰度，结果灰度阶段问题频出
- 原因：将灰度当作测试手段而非发布策略
- 对策：灰度是最后一道防线而非第一道，上线前测试必须充分
