# 文档能力

> 所属板块: 软技能 | 编号: P04 | 最后更新: 2026-03-29

---

## 1. 概述

文档能力是QA专业性的重要体现，也是团队知识沉淀和传承的基础。优秀的文档能力不仅包括"写得好"，更包括"写得对"（内容准确）、"写得清"（结构清晰）、"写得用"（有实际价值）。在游戏QA工作中，测试计划、测试报告、用例评审记录、复盘总结、知识库文章等文档贯穿整个测试生命周期，直接影响团队的工作效率和质量水平。

**核心价值**：
- 测试计划确保测试工作有序、可控、可追溯
- 测试报告为上线决策提供客观依据
- 用例评审提升测试覆盖的完整性和有效性
- 复盘总结推动团队持续改进
- 知识库建设降低人员流动带来的知识损失

## 2. 知识体系

### 2.1 测试计划编写规范（参考IEEE 829）

**测试计划核心要素**：

| 要素 | 内容 | 说明 |
|------|------|------|
| 测试范围 | 测试什么/不测什么 | 明确边界，避免遗漏和越界 |
| 测试策略 | 怎么测/用什么方法 | 功能测试/性能测试/兼容性测试等策略 |
| 测试环境 | 在什么环境测 | 服务器/客户端/设备/网络环境 |
| 测试资源 | 谁来测/需要什么支持 | 人员分工/工具/测试数据/GM权限 |
| 测试进度 | 什么时候测/里程碑 | 各阶段时间节点和交付物 |
| 准入/准出标准 | 什么条件开始/结束测试 | 量化的质量门禁 |
| 风险评估 | 有什么风险/如何应对 | 识别的风险和应对策略 |
| 交付物 | 产出什么文档/报告 | 测试报告/Bug清单/风险清单等 |

**测试计划编写原则**：
1. 可执行性：计划要具体到可操作，不写空话
2. 可度量性：目标和标准要量化
3. 适应性：预留变更空间，不过度僵化
4. 简洁性：突出重点，避免冗长

### 2.2 测试报告要素

**测试报告结构**：

| 章节 | 内容 | 关键信息 |
|------|------|----------|
| 概述 | 版本信息/测试周期/测试范围 | 让读者快速了解背景 |
| 测试执行情况 | 用例执行数/通过率/覆盖率 | 量化的测试进度 |
| Bug统计分析 | Bug数量/分布/趋势/修复率 | 质量状态的客观反映 |
| 遗留问题 | 未修复Bug清单及风险评估 | 上线决策的关键依据 |
| 风险评估 | 已识别风险及应对状态 | 帮助决策者了解风险全貌 |
| 质量结论 | 放行建议/附加条件 | 明确的质量判断 |
| 附录 | 详细数据/测试环境/工具版本 | 补充信息 |

**Bug统计维度**：
- 按严重程度分布（P0/P1/P2/P3）
- 按模块分布（哪个模块Bug最多）
- 按状态分布（新建/修复中/已修复/已验证/关闭）
- 按趋势变化（每日新增/修复/遗留曲线）
- 按类型分布（功能/性能/兼容/UI/崩溃）

### 2.3 用例评审流程

**用例评审目标**：
- 验证用例覆盖的完整性（是否覆盖所有需求点）
- 验证用例设计的有效性（是否能发现问题）
- 验证用例描述的准确性（是否可执行）
- 发现遗漏的测试场景

**评审参与角色**：

| 角色 | 评审关注点 |
|------|------------|
| QA（编写者） | 讲解用例设计思路和覆盖范围 |
| QA（评审者） | 检查覆盖完整性、用例有效性 |
| 策划 | 确认需求理解是否正确、是否有遗漏场景 |
| 开发 | 补充技术实现相关的测试点（边界/异常/并发） |

**用例评审检查维度**：

| 维度 | 检查点 |
|------|--------|
| 完整性 | 需求点是否100%覆盖？正常/异常/边界是否都有？ |
| 有效性 | 用例是否能有效发现问题？是否有冗余用例？ |
| 准确性 | 前置条件/操作步骤/预期结果是否准确？ |
| 可执行性 | 其他人能否按用例独立执行？ |
| 优先级 | 用例优先级是否合理？核心场景是否标记为高优？ |

### 2.4 复盘方法论

**PDCA循环**：

| 阶段 | 含义 | QA实践 |
|------|------|--------|
| Plan（计划） | 制定目标和计划 | 测试计划/测试策略 |
| Do（执行） | 按计划执行 | 测试执行/Bug跟踪 |
| Check（检查） | 检查执行结果 | 测试报告/数据分析 |
| Act（改进） | 总结经验，持续改进 | 复盘总结/流程优化 |

**AAR（After Action Review）复盘法**：

| 步骤 | 关键问题 | 输出 |
|------|----------|------|
| 回顾目标 | 我们原计划做什么？ | 目标清单 |
| 评估结果 | 实际发生了什么？ | 事实描述（时间线） |
| 分析原因 | 为什么会这样？（好的和不好的） | 根因分析 |
| 总结经验 | 我们学到了什么？ | 经验教训清单 |
| 制定行动 | 下次如何改进？ | 改进行动项（含负责人和截止日期） |

**复盘常见主题**：
- 版本复盘：每个版本结束后的质量复盘
- 事故复盘：线上事故的根因分析和改进
- 项目复盘：项目里程碑或结项时的全面复盘
- 流程复盘：定期审视测试流程的有效性

### 2.5 知识管理体系

**知识分类**：

| 类别 | 内容示例 | 维护频率 |
|------|----------|----------|
| 项目知识 | 系统架构/模块说明/接口文档 | 随版本更新 |
| 测试知识 | 测试方法/工具使用/最佳实践 | 持续积累 |
| 业务知识 | 品类特征/竞品分析/行业趋势 | 定期更新 |
| 踩坑经验 | Bug案例/事故复盘/常见问题 | 随时记录 |
| 流程规范 | 测试流程/编码规范/评审标准 | 定期审视 |

**知识库建设原则**：
1. 可搜索：良好的分类和标签体系
2. 可维护：明确的维护责任人和更新机制
3. 可复用：内容模板化、结构化
4. 有价值：定期清理过时内容，保持知识库的时效性

**知识沉淀机制**：
```
日常沉淀:
├── Bug案例库: 有代表性的Bug记录分析过程和根因
├── 踩坑记录: 每次踩坑及时记录到经验库
└── 工具技巧: 发现好用的工具/技巧及时分享

定期沉淀:
├── 版本复盘: 每个版本的质量数据和经验教训
├── 技术分享: 每月1次团队内部技术分享
└── 知识审计: 每季度审视知识库内容时效性

专项沉淀:
├── 新人手册: 新人入职所需的核心知识汇总
├── 最佳实践: 经过验证的测试方法和流程
└── 工具手册: 团队使用的工具操作指南
```

## 4. 实践框架/检查清单

### 4.1 测试计划编写检查清单

- [ ] 明确测试范围（测什么/不测什么）
- [ ] 制定测试策略（各类型测试的方法和工具）
- [ ] 明确测试环境需求并确认可用性
- [ ] 制定测试进度计划（含里程碑和交付物）
- [ ] 明确人员分工和职责
- [ ] 定义准入/准出标准（量化指标）
- [ ] 完成风险评估并制定应对策略
- [ ] 计划已与项目组对齐并获得确认

### 4.2 测试报告编写检查清单

- [ ] 版本信息和测试周期准确
- [ ] 测试执行数据完整（用例数/通过率/覆盖率）
- [ ] Bug统计数据准确且多维度分析
- [ ] 遗留问题清单完整（含风险评估和规避方案）
- [ ] 质量结论明确（放行/有条件放行/不放行）
- [ ] 数据图表清晰易读
- [ ] 报告已经过自审和同行审查

### 4.3 用例评审检查清单

- [ ] 用例覆盖所有需求点（需求追溯矩阵）
- [ ] 包含正常流程、异常流程、边界条件用例
- [ ] 用例步骤清晰、可执行
- [ ] 预期结果明确、可验证
- [ ] 用例优先级标注合理
- [ ] 评审意见已记录并跟踪闭环
- [ ] 评审后用例已更新并再次确认

### 4.4 复盘文档检查清单

- [ ] 复盘目标明确（复盘什么/期望产出什么）
- [ ] 事实描述客观准确（有数据支撑）
- [ ] 根因分析深入（不停留在表面原因）
- [ ] 经验教训有总结（好的继续/不好的改进）
- [ ] 改进行动项具体可执行（含负责人和截止日期）
- [ ] 改进项有跟踪机制确保落地

### 4.5 知识库文章编写检查清单

- [ ] 标题清晰，能准确反映内容
- [ ] 分类和标签正确
- [ ] 内容结构清晰（问题→分析→解决方案）
- [ ] 关键步骤有截图或代码示例
- [ ] 适用范围和前置条件说明清楚
- [ ] 内容经过验证，确保准确性
- [ ] 标注作者和更新日期

## 5. 执行流程

### 5.1 测试计划编写SOP

```
Step 1: 信息收集
├── 阅读需求文档和技术方案
├── 了解版本目标和时间节点
├── 确认可用的测试资源
└── 收集历史版本的质量数据

Step 2: 计划编写
├── 确定测试范围和策略
├── 制定测试进度计划
├── 分配人员和职责
├── 定义准入/准出标准
├── 完成风险评估
└── 按模板编写测试计划文档

Step 3: 计划评审
├── 内部自审（检查完整性和合理性）
├── 同行评审（QA同事交叉审查）
├── 项目组评审（策划/开发/PM确认）
└── 根据评审意见修改完善

Step 4: 计划执行与更新
├── 按计划执行测试
├── 跟踪进度偏差
├── 需求变更时及时更新计划
└── 版本结束后归档
```

### 5.2 测试报告编写SOP

```
Step 1: 数据收集
├── 导出用例执行数据
├── 导出Bug统计数据
├── 收集性能/兼容性测试数据
└── 整理遗留问题清单

Step 2: 数据分析
├── 计算关键指标（通过率/覆盖率/修复率）
├── 分析Bug分布趋势
├── 评估遗留问题的风险等级
└── 制作数据图表

Step 3: 报告编写
├── 按模板填写各章节
├── 撰写质量结论和放行建议
├── 附上支撑数据和图表
└── 自审检查数据准确性

Step 4: 报告发布
├── 同行审查
├── 发送给项目组
├── 在上线评审会上汇报
└── 归档存储
```

### 5.3 用例评审SOP

```
Step 1: 评审准备
├── 用例编写完成并自审
├── 提前1天发送用例给评审人
├── 预约评审会议（30-60min）
└── 准备需求追溯矩阵

Step 2: 评审执行
├── 编写者讲解用例设计思路（10min）
├── 按模块逐项评审（30-40min）
│   ├── 检查覆盖完整性
│   ├── 检查用例有效性
│   ├── 补充遗漏场景
│   └── 优化用例描述
└── 总结评审结论（5min）

Step 3: 评审跟进
├── 整理评审意见清单
├── 按意见修改用例
├── 修改后发送确认
└── 归档评审记录
```

### 5.4 复盘执行SOP

```
Step 1: 复盘准备
├── 确定复盘主题和范围
├── 收集相关数据和事实
├── 邀请相关参与者
└── 准备复盘模板

Step 2: 复盘会议（60-90min）
├── 回顾目标: 我们原计划做什么？（10min）
├── 评估结果: 实际发生了什么？（15min）
├── 分析原因: 为什么？（30min）
│   ├── 做得好的 → 继续保持
│   └── 做得不好的 → 根因分析
├── 制定行动: 如何改进？（20min）
└── 总结确认（5min）

Step 3: 复盘跟进
├── 整理复盘报告并发送
├── 将改进项录入跟踪系统
├── 定期检查改进项进展
└── 下次复盘时回顾上次改进项落地情况
```

## 6. 能力评估指标

| 等级 | 文档能力表现 |
|------|--------------|
| L0-入门 | 能按模板填写基本的测试文档；文档结构基本完整但内容较粗糙；Bug描述基本清晰；能参与用例评审但较少提出意见 |
| L1-基础 | 能独立编写测试计划和测试报告；用例设计覆盖正常流程和主要异常流程；能记录评审意见并跟踪闭环；能按模板编写复盘文档 |
| L2-进阶 | 测试计划策略合理、风险评估到位；测试报告数据分析深入、结论有说服力；用例评审能发现覆盖遗漏并提出改进建议；复盘分析能触及根因并产出有效改进项；开始建设个人知识库 |
| L3-专业 | 能制定团队级的文档规范和模板；测试报告被管理层作为决策依据；能主导用例评审并提升团队用例质量；复盘产出的改进项落地率高且效果显著；主导团队知识库建设 |
| L4-专家 | 能建立完整的测试文档体系和知识管理体系；文档规范被其他团队参考借鉴；能培养团队成员的文档能力；知识库成为团队核心资产，有效降低人员流动影响；在行业内有文档/知识管理的实践分享 |

## 7. 踩坑经验库

### 高频踩坑 TOP 5

**坑1: 测试计划写了不更新，成为摆设**
- 场景：版本初期写了测试计划，后续需求变更多次但计划未更新
- 原因：把测试计划当作一次性文档而非活文档
- 对策：需求变更时同步更新测试计划，至少更新测试范围和进度部分

**坑2: 测试报告只有数据没有分析**
- 场景：报告罗列了大量数据，但没有分析和结论，决策者看不懂
- 原因：把报告当作数据汇总而非决策支持工具
- 对策：报告必须有"So What"——数据说明了什么问题？建议是什么？

**坑3: 用例评审走过场，评审意见不落地**
- 场景：评审会上提了很多意见，但会后没人跟进修改
- 原因：评审意见没有记录和跟踪机制
- 对策：评审意见必须记录到文档/系统中，指定负责人和截止日期，下次评审前检查

**坑4: 复盘只找问题不找根因，改进项空泛**
- 场景：复盘总结"以后要注意"，但下次同样的问题还是发生
- 原因：根因分析不深入，改进项不具体
- 对策：使用5Why法深挖根因，改进项必须具体到"谁在什么时间做什么事"

**坑5: 知识库建了没人用，逐渐荒废**
- 场景：花大力气建了知识库，但团队成员不查阅也不更新
- 原因：知识库内容不实用/不好找/不更新
- 对策：知识库内容要解决实际问题；建立良好的分类和搜索；将知识库使用融入日常流程（如新人入职必读、Bug分析必查）
